# Business Requirement: Analytics Page Redesign

## Executive Summary

The Analytics page (`/app/attribution`) currently displays UTM attribution data under a subtitle that promises "bundle revenue" data which doesn't exist — this misleads merchants and undersells the real revenue data we do have. This redesign merges existing UTM attribution data with a new Bundle Revenue section powered by the already-populated `OrderAttribution` table, presenting a cohesive analytics dashboard that merchants actually trust and use.

---

## Problem Statement

**Who experiences it:** Every merchant who visits the Analytics page.

**Pain points:**
1. **False promise**: The page subtitle says "UTM attribution & bundle revenue" but there is no bundle revenue section. Merchants who upgrade to check their ROI find nothing.
2. **Buried signal**: The `OrderAttribution` table already stores per-bundle revenue (orders matched to bundles via product ID). This data is never shown in a "bundle revenue" context — it's only surfaced as a footnote inside the UTM attribution section.
3. **Weak first impression**: The page looks like a dev stub. Empty states are uninformative. There's no hierarchy of what to look at first, no hero KPIs, and no action-oriented layout.
4. **Analytics page not gated**: Free plan users see the same page as Grow plan users, yet the pricing page says "bundle analytics coming soon" as a Grow feature — contradiction that erodes trust.

**Business impact:**
- Merchants who don't see value from analytics don't upgrade
- Merchants who see the false "bundle revenue" subtitle feel deceived
- The existing revenue data we have is wasted — it should be the primary value prop

---

## Research Findings

### What data exists today

**`OrderAttribution` table (production data):**
- Revenue per order in cents (`revenue`, `currency`)
- Bundle association (`bundleId` — nullable; non-null when order contained a bundle product)
- UTM campaign data (`utmSource`, `utmMedium`, `utmCampaign`)
- Order metadata (`orderId`, `orderNumber`, `landingPage`)
- Timestamps (`createdAt`)
- Indexes optimized for time-windowed queries per shop and per bundle

**`Bundle` table:**
- Bundle name, type, status
- `OrderAttribution[]` relation (already defined)

**`BundleAnalytics` table:**
- Schema exists for `view`, `add_to_cart`, `purchase` events
- **Zero rows — never populated**
- Widget fires no analytics events today

### What data is missing for funnel metrics

| Metric | Blocked by | Effort to unblock |
|--------|-----------|------------------|
| Bundle views / impressions | Widget fires no view events | Medium — add JS event + API endpoint |
| Add-to-cart rate | Widget fires no click events | Medium — same as above |
| Cart abandonment | Requires session tracking | High |
| Variant / product performance | Not tracked at checkout | High |

## Recommended Approach

### Option A — Redesign page using existing `OrderAttribution` data only (Recommended MVP)

Use only the data that already exists and is already populated. Show:
1. **Hero KPI bar** — Total Bundle Revenue, Bundle Orders, Bundle AOV, % of orders from bundles
2. **Revenue by bundle** — leaderboard card: each active bundle, its revenue, order count, AOV
3. **Revenue trend chart** — time-series of bundle revenue vs non-bundle revenue (daily/weekly)
4. **UTM attribution section** — existing section, cleaned up visually

Fix the subtitle to "Bundle revenue & UTM attribution" (swap order to match importance). No schema changes, no new data collection.

**Pros:** Ships immediately. Real data. No instrumentation risk. No DB changes.
**Cons:** No funnel/conversion data (views, CTR). Can't show conversion rate yet.

### Option B — Add widget event tracking to unlock funnel metrics

Instrument the widgets to fire `view` and `add_to_cart` events to a new `/api/analytics/event` endpoint, writing to the existing `BundleAnalytics` table.

**Pros:** Enables conversion rate, funnel analysis, full attribution story.
**Cons:** Requires widget JS changes + new API endpoint + DB backfill. Takes longer. Risks widget regression.

### Option C — Third-party analytics integration (Amplitude, Segment, etc.)

**Cons:** Cost, complexity, data leaving our infra. Out of scope for a Shopify app.

**Verdict:** ✅ Option A as MVP, ✅ Option B as a follow-on track documented separately.

## Functional Requirements

- **FR-01:** The page subtitle must accurately describe its contents: `"Bundle revenue & UTM attribution"`
- **FR-02:** A hero KPI row must show at-a-glance figures: Total Bundle Revenue, Bundle Orders, Bundle AOV, % Revenue from Bundles
- **FR-03:** All KPIs must respond to the existing time-range selector (7 / 30 / 90 days)
- **FR-04:** A "Revenue by Bundle" card must list each bundle with its attributed revenue, order count, and AOV for the selected period
- **FR-05:** The revenue trend chart must be able to show "bundle revenue" as its primary series (not just total attributed revenue)
- **FR-06:** The UTM attribution section (platform/medium/campaign/landing page breakdowns) must remain, visually subordinated below the bundle revenue section
- **FR-07:** Empty state: when no `OrderAttribution` data exists, show an informative empty state (not a blank page) explaining how to generate data (enable pixel + tag ads)
- **FR-08:** Period-over-period comparison on hero KPIs (e.g. "+23% vs last period")

---

## Non-Functional Requirements

- **NFR-01:** Page must load in <2s for shops with ≤10k `OrderAttribution` rows
- **NFR-02:** No new DB columns or migrations required for MVP
- **NFR-03:** The redesign must not break the existing attribution API or data pipeline
- **NFR-04:** All existing UTM attribution data is preserved and visible

---

## User Stories (high level)

- As a merchant, I want to see my bundle revenue on the Analytics page so that I can prove the ROI of my bundles to myself and my team
- As a merchant, I want to see which bundle generates the most revenue so that I know where to focus my marketing spend
- As a merchant, I want to see a revenue trend chart that shows bundle vs non-bundle orders so I understand what percentage of my revenue comes from bundles
- As a merchant, I want KPIs to compare this period vs last period so I can tell if my bundles are growing

---

## Scope

### In Scope (MVP)
- Visual redesign of the full Analytics page
- Hero KPI bar with 4 metrics + period-over-period
- Bundle revenue leaderboard card
- Revenue trend chart (bundle revenue series)
- Fixed subtitle
- Improved empty state

### Out of Scope (Follow-on)
- Widget event tracking (views, add-to-cart) → `BundleAnalytics` table
- Conversion funnel visualization
- Plan-gating analytics features
- Export / CSV download
- Shopify Audiences or paid attribution integrations

---

## Risks

| Risk | Likelihood | Mitigation |
|------|-----------|------------|
| `OrderAttribution.revenue` is stored in cents — double-counting if displayed raw | Low | Already handled in existing page; apply same `/ 100` conversion |
| Some shops have zero `OrderAttribution` rows (pixel not enabled) | High | Robust empty state with setup instructions |
| Bundle names may be long or many — leaderboard card overflow | Medium | Truncate names + tooltip; cap list at 10, add "View all" |
| Period-over-period comparison requires two DB queries | Low | Already done in existing loader; extend pattern |
