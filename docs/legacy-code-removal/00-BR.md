# Business Requirement: Legacy Code Removal

## Executive Summary

Remove dead and deprecated legacy code from the codebase that has zero active callers, reducing maintenance surface area and eliminating confusion for future developers.

## Problem Statement

The codebase contains multiple layers of deprecated code that were intentionally left in place during earlier refactors for "backwards compatibility" or "safe migration". A full audit reveals that several of these are now genuinely dead — no callers, no routes reference them, and they are only re-exported through index files that are themselves never imported for these symbols. This dead code increases cognitive load, adds false import surface area, and creates a misleading picture of what the system actually does.

## Research Findings

A grep audit of the entire codebase (`/app`, `/extensions`) identified the following categories:

### Category A — Dead Code (zero callers anywhere)

| Item | File | Evidence |
|------|------|----------|
| `widget-installation-legacy.server.ts` | `app/services/widget-installation/` | Entire file marked `@deprecated`, zero callers in app/routes/extensions |
| Legacy static method re-exports (`checkWidgetInstallation`, `checkFullPageWidgetInstallation`) | `widget-installation-core.server.ts` lines 99–106 | Zero callers |
| Legacy exports from `widget-installation/index.ts` | `index.ts` lines 34–39 | Zero callers |
| `ensureBundleMetafieldDefinitions()` wrapper | `definitions.server.ts` line 166 | Re-exported in index files but zero callers in routes/services |
| `getStepSelectionText()` method | `bundle-widget-product-page.js` line 823 | Comment says "no longer used", zero invocations in file |

### Category B — Active Backwards Compatibility (KEEP)

| Item | File | Reason to Keep |
|------|------|----------------|
| Checkout component legacy object format | `Checkout.tsx` | Supports in-flight carts with old data format |
| Widget messaging `pricing.messages` fallback | both widget JS files | Existing saved bundles use old metafield path |
| `legacyResourceId` in GraphQL queries | both configure routes | Numeric ID still needed for admin product URL |

### Recommended Approach

**Remove Category A only.** Category B items are still actively guarding against runtime failures for existing merchant data and should not be touched.

## Functional Requirements

- FR-01: Delete `widget-installation-legacy.server.ts`
- FR-02: Remove the legacy static method re-exports from `widget-installation-core.server.ts`
- FR-03: Remove the legacy exports from `widget-installation/index.ts`
- FR-04: Remove the `ensureBundleMetafieldDefinitions` wrapper function and its re-exports from both index files
- FR-05: Remove the `getStepSelectionText()` dead method from `bundle-widget-product-page.js`

## Non-Functional Requirements

- NFR-01: Zero runtime behaviour change — no merchant-facing functionality is affected
- NFR-02: TypeScript must compile without new errors after removal
- NFR-03: All existing tests must continue to pass

## Scope

### In Scope
- Deletion of the legacy widget installation file
- Removal of dead wrapper functions and re-exports
- Removal of the unreachable widget method

### Out of Scope
- Active backwards-compatibility code (Category B)
- Any test files (tests covering legacy data formats remain valid)
- Documentation/archive files referencing removed functions

## Risks

| Risk | Likelihood | Mitigation |
|------|------------|------------|
| Hidden dynamic import of legacy methods | Low | Grep confirms zero string references to these function names |
| TypeScript barrel re-export breaks a type import | Low | Only function exports are removed; no type-only exports in scope |
| Widget rebuild not run after JS change | Medium | Run `npm run build:widgets` in SDE phase |
