# Business Requirement: Pub/Sub → Direct Render Webhook Migration

## Executive Summary

The Google Cloud Pub/Sub free trial has expired, cutting off the webhook ingestion layer that bridges Shopify events to the app's processing logic. The existing Render worker service must be converted from a Pub/Sub subscriber to a direct HTTP webhook receiver so Shopify delivers events without any GCP dependency.

## Problem Statement

**Who is affected:** The app itself — webhook-driven business logic (subscription downgrades, product draft-on-archive, GDPR compliance, app uninstall cleanup) has silently stopped executing since Pub/Sub became unavailable.

**What breaks:**
- Billing webhooks (`app_subscriptions/update`, `app_subscriptions/cancelled`) — merchants not downgraded correctly when they cancel
- Product lifecycle webhooks (`products/update`, `products/delete`) — bundles not drafted/archived when component products change
- GDPR compliance webhooks (`customers/data_request`, `customers/redact`, `shop/redact`) — legal obligation unmet
- App lifecycle webhooks (`app/uninstalled`, `app/scopes_update`) — cleanup and session updates not running


## Research Findings

### Approach A: GCP EventBridge / Alternative Cloud Queue
- Replace GCP Pub/Sub with AWS EventBridge or another managed queue
- **Pros:** Retains async decoupling; built-in retry queuing
- **Cons:** New paid infrastructure; another vendor; migration effort without business value; still requires Shopify webhook URI as `arn:aws:...`
- **Verdict:** ❌ Rejected — introduces cost and complexity with no new capability

### Approach B: Route Shopify Webhooks to Main Remix App
- Change `shopify.app.toml` to point webhooks at the main app (`/api/webhooks`)
- Shopify's `authenticate.webhook()` SDK handles HMAC + idempotency
- **Pros:** Uses Shopify SDK's built-in webhook handling; simplest TOML change
- **Cons:** Mixes admin UI requests with webhook processing load on the same Remix dyno; makes the Render worker service redundant/idle (wasting spend); requires refactor of processing logic to Shopify SDK handler style
- **Verdict:** ❌ Rejected — defeats the purpose of the separate worker; processing logic refactor is unnecessary churn

### Approach C: Direct HTTP Webhook Receiver on Render Worker ✅ Recommended
- Replace the Pub/Sub polling loop in the Render worker with a lightweight Node.js HTTP server
- Shopify POSTs webhook payloads directly to `https://wolfpack-production-pubsub-worker.onrender.com/webhooks`
- Worker validates Shopify HMAC signature, adapts the raw HTTP payload to the existing `PubSubMessage` format, and calls `WebhookProcessor.processPubSubMessage()` directly (same module call as today)
- Update `shopify.app.toml` to use HTTPS URIs instead of `pubsub://`
- **Pros:** Zero change to processing logic; eliminates GCP entirely; same Render worker service; HMAC validation improves security over current IAM-only approach; Render health-check `/health` can be added for free
- **Cons:** Worker must stay alive (no sleep on free tier) to receive webhooks — but this is already the case in production

## Functional Requirements

- **FR-01:** The Render worker must expose a `POST /webhooks` HTTP endpoint that accepts Shopify webhook POSTs.
- **FR-02:** The endpoint must validate the `X-Shopify-Hmac-Sha256` header using `SHOPIFY_API_SECRET` before processing any payload.
- **FR-03:** All 11 currently-handled webhook topics must continue to work after migration: `app_subscriptions/update`, `app_subscriptions/cancelled`, `app_subscriptions/approaching_capped_amount`, `app_purchases_one_time/update`, `products/update`, `products/delete`, `customers/data_request`, `customers/redact`, `shop/redact`, `app/uninstalled`, `app/scopes_update`.
- **FR-04:** The worker must return HTTP 200 immediately after queuing/starting processing (never block Shopify > 5 s).
- **FR-05:** `shopify.app.toml` (production) and `shopify.app.wolfpack-product-bundles-sit.toml` (staging) must be updated to declare HTTPS webhook URIs pointing to the Render worker.
- **FR-06:** The `@google-cloud/pubsub` package and all GCP environment variables must be removed.
- **FR-07:** A `GET /health` endpoint must be added to the worker for Render health checks.
- **FR-08:** Existing idempotency logic in `WebhookProcessor` must remain intact (no change).

## Non-Functional Requirements

- **NFR-01 Security:** HMAC validation must use constant-time comparison. Any request with an invalid or missing signature must return 401 and be logged.
- **NFR-02 Reliability:** Worker process must handle SIGTERM gracefully (close HTTP server, wait for in-flight requests).
- **NFR-03 Observability:** All received webhooks must be logged with topic + shop domain. Failed HMAC checks must be logged at `error` level.
- **NFR-04 Backward Compatibility:** No change to `WebhookProcessor`, handlers, or database schema. The `PubSubMessage` interface is reused as an internal adapter type.
- **NFR-05 Startup:** Worker must bind to `process.env.PORT` (Render injects this automatically).

## User Stories (high level)

- As **the app**, I want Shopify webhook events to reach `WebhookProcessor` so that billing, product, GDPR, and lifecycle logic executes reliably.
- As **a developer**, I want the Render worker to receive webhooks directly so that we have no GCP dependency or cost.
- As **Shopify's platform**, I want to deliver webhooks to an HTTPS endpoint that responds within 5 seconds and returns 200 so that retries are not triggered.

## Scope

### In Scope
- Replace `scripts/pubsub-worker.ts` + `app/services/pubsub-worker.server.ts` with a new HTTP worker
- New script entry point `scripts/webhook-worker.ts`
- New service `app/services/webhook-worker.server.ts`
- Update both `shopify.app.toml` files
- Remove `@google-cloud/pubsub` from `package.json`
- Remove GCP env vars from documentation/`.env.example`
- Update Render worker service start command

### Out of Scope
- Changes to `WebhookProcessor` or any handler
- Changes to the main Remix app (no new routes needed)
- Re-registering webhooks via Shopify Admin GraphQL API (TOML + `shopify app deploy` handles this)
- Adding a message queue / retry buffer (idempotency at DB level is sufficient)

## Risks

| Risk | Likelihood | Mitigation |
|------|------------|------------|
| Shopify retries during deploy window | Medium | Keep old Pub/Sub topic alive briefly; deploy worker first, then `shopify app deploy` |
| Render worker sleeps (free tier) and misses a webhook | Low (production uses paid plan) | Render `/health` endpoint keeps service alive |
| HMAC secret mismatch between envs | Low | Validate `SHOPIFY_API_SECRET` at worker startup |
| Wrong Render worker URL in TOML | Low | Confirm URL from Render dashboard before committing |
