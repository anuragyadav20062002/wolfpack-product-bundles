# Business Requirement: Feed-Ready Product Enhancement (Ad-Ready Phase 1)

## Executive Summary

Wolfpack bundle products are invisible to advertising platforms because they have $0 price and unmanaged inventory. This phase fixes both, making bundles eligible for Google Shopping, Meta Catalog, and TikTok Shop feeds via Shopify's native sales channel sync.

## Problem Statement

**Who:** Shopify merchants using Wolfpack to create product bundles who want to advertise those bundles on Google Shopping, Meta (Facebook/Instagram), and TikTok.

**Pain point:** When a merchant publishes a Wolfpack bundle to an ad channel (e.g., Google & YouTube), the bundle is rejected or hidden because:

1. **Price is $0.00** — Ad platforms require a valid, non-zero price. Google Merchant Center explicitly rejects products with `price: 0`. Currently, bundle parent products are created with `price: "0.00"` in two locations:
   - `handlers.server.ts` (dashboard create/clone)
   - `app.bundles.cart-transform.tsx` (cart transform route)

2. **Inventory is unmanaged** — Bundle parent variants have `inventoryManagement: null` and `inventoryPolicy: "DENY"`. Ad platforms interpret unmanaged inventory as unknown availability, which may prevent the product from appearing in feeds or showing "out of stock".

3. **No inventory sync** — Even if inventory management is enabled, there is no mechanism to calculate and maintain bundle inventory based on component product stock levels.

**Why it matters:** Wolfpack's key differentiator is creating **real Shopify products** for bundles (vs. virtual bundles). But that advantage is wasted if the products fail ad platform eligibility checks. Fixing this unlocks the entire ad-ready value proposition.

## Research Findings

### Current Bundle Product Creation (Verified)

| Setting | Current Value | Required Value | Location |
|---------|--------------|----------------|----------|
| `price` | `"0.00"` | Calculated bundle price | `handlers.server.ts:199`, `cart-transform.tsx:195` |
| `inventoryManagement` | `null` | `"SHOPIFY"` | `handlers.server.ts:200`, `cart-transform.tsx:197` |
| `inventoryPolicy` | `"DENY"` | `"DENY"` (keep — bundle is a container) | Same locations |

### Price Calculation Already Exists

`calculateBundlePrice()` in `pricing-calculation.server.ts` (line 237) already computes the correct display price:
- Iterates bundle steps, averages product prices per step
- Applies discount (percentage_off currently supported in this function)
- Returns price as string with 2 decimal places
- `updateBundleProductPrice()` (line 319) updates the Shopify variant

**The price calculation code exists but is NOT called during product creation.** It's only called during metafield sync (configure page save). Bundles created but never synced retain `$0.00`.

### Cart Transform Impact Analysis

| Cart Transform Path | Uses Parent Variant Price? | Impact of Price Change |
|--------------------|--------------------------|-----------------------|
| **MERGE** (component products merged) | NO — always uses `percentageDecrease` calculated from component sum | **SAFE** — no behavior change |
| **EXPAND, discount > 0%** | NO — uses `percentageDecrease` | **SAFE** — no behavior change |
| **EXPAND, discount = 0%** | YES — no price override applied, Shopify uses variant price | **Bug fix** — changes from $0 to actual price (correct behavior) |

### Inventory Calculation Approach

**Formula:** `bundle_stock = MIN(component_inventory[i] / component_quantity[i])`

This is the industry-standard approach (used by Shopify's own native bundles). It accounts for bundles that require multiple units of a component.

**Trigger events for recalculation:**
- `inventory_levels/update` webhook (not currently subscribed)
- Bundle creation/update in Wolfpack
- Periodic reconciliation (optional, for data integrity)

### Approach A: Managed Inventory with Sync Engine

**Description:** Enable `inventoryManagement: "SHOPIFY"` on bundle parent variants. Build a service that calculates bundle inventory from component levels and sets it via `inventoryAdjustQuantities`.

**Pros:**
- Ad platforms see real inventory counts and correct availability
- Bundles show "in stock" / "out of stock" accurately
- Standard Shopify inventory workflow (locations, adjustments, audit trail)

**Cons:**
- Requires new scopes (`write_inventory`, `read_inventory`) — forces merchant re-auth
- Webhook processing load for `inventory_levels/update`
- Must handle edge cases (multi-location, untracked components)

### Approach B: Keep Unmanaged, Set inventoryPolicy to CONTINUE

**Description:** Keep `inventoryManagement: null` but change `inventoryPolicy` to `"CONTINUE"` so the product always appears purchasable.

**Pros:**
- No new scopes needed
- Simpler implementation

**Cons:**
- Ad platforms may still treat unmanaged inventory as unknown
- No accurate stock representation — bundles could appear "in stock" when components are sold out
- Google Merchant Center may flag availability mismatch between feed and landing page

### **Recommended:** Approach A — Managed Inventory with Sync Engine

Approach A is the only option that provides accurate feed data. Approach B risks ad platform disapprovals and poor merchant experience (customers clicking ads for out-of-stock bundles).

## Functional Requirements (FRs)

- **FR-01:** When a bundle product is created in Shopify, the variant price MUST be set to the calculated bundle price (not $0.00).
- **FR-02:** When a bundle product is created in Shopify, the variant MUST have `inventoryManagement: "SHOPIFY"` to enable inventory tracking.
- **FR-03:** Wolfpack MUST calculate bundle inventory as `MIN(component_inventory[i] / component_quantity[i])` for all component variants.
- **FR-04:** Wolfpack MUST set the calculated inventory quantity on the bundle parent variant via the Shopify Inventory API.
- **FR-05:** When any component product's inventory changes, Wolfpack MUST recalculate and update the bundle's inventory (webhook-driven).
- **FR-06:** Existing bundle products with $0 price and unmanaged inventory MUST be migrated to correct values via a batch migration.
- **FR-07:** The inventory sync engine MUST handle edge cases: components with no inventory tracking (treat as unlimited), MIN = 0 (set bundle as out of stock), multi-location stores (use primary location for Phase 1).

## Non-Functional Requirements (NFRs)

- **NFR-01:** Performance — Inventory recalculation for a single bundle MUST complete within 5 seconds (API round-trips to Shopify).
- **NFR-02:** Backward compatibility — MERGE cart transform path MUST NOT change behavior. EXPAND path with discount MUST NOT change behavior.
- **NFR-03:** Reliability — Webhook-driven inventory updates MUST be idempotent (safe to replay).
- **NFR-04:** Scope batching — All new OAuth scopes (`write_inventory`, `read_inventory`, `read_orders`) MUST be deployed in a single `shopify app deploy` to minimize merchant re-auth to one cycle. (`read_orders` is included early for Phase 3.)

## User Stories (high level)

- As a **merchant**, I want my bundle products to have a real price so that they appear correctly in Google Shopping and Meta Catalog feeds.
- As a **merchant**, I want my bundle products to show accurate inventory levels so that ad platforms display correct availability.
- As a **merchant**, I want bundle inventory to update automatically when I restock or sell component products so that ads stay accurate without manual intervention.
- As a **developer**, I want existing bundles to be migrated to correct price and inventory values so that merchants don't need to recreate bundles.

## Scope

### In Scope
- Fix bundle variant price on creation (both dashboard and cart-transform paths)
- Enable `inventoryManagement: "SHOPIFY"` on new bundles
- Build inventory sync engine (`app/services/bundles/inventory-sync.server.ts`)
- Register and handle `inventory_levels/update` webhook
- Batch migration for existing bundle products
- Add `write_inventory`, `read_inventory`, `read_orders` scopes to `shopify.app.toml`

### Out of Scope
- Multi-channel publication (Phase 2)
- Campaign/UNLISTED bundles (Phase 2)
- UTM attribution (Phase 3)
- Multi-location inventory (use primary location only for Phase 1)
- Real-time inventory display on storefront widget (out of scope entirely)
- Compare-at price / "save X%" display in feeds

## Risks

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Merchant re-auth friction from new scopes | HIGH (guaranteed) | MEDIUM | Batch all scopes in one deploy; communicate via in-app banner |
| EXPAND bundles with 0% discount show different price | LOW (uncommon config) | LOW | This is a bug fix; test thoroughly before deploy |
| Inventory webhook volume overwhelms processor | MEDIUM | MEDIUM | Debounce recalculation per bundle (skip if recalculated <60s ago) |
| Component product has no inventory tracking | MEDIUM | LOW | Treat as unlimited stock (do not constrain bundle) |
| Migration fails for some bundles (missing Shopify product) | LOW | LOW | Log failures, skip gracefully, report to admin |
