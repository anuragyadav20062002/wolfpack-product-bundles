# Business Requirement: Admin UI Tier Configuration for Full-Page Bundle Widget

## Executive Summary

Merchants using the full-page bundle widget currently must enter pricing tier labels and bundle IDs directly in the Shopify Theme Editor block settings. This feature moves that configuration into the Remix admin app, removing the need for merchants to ever open the Theme Editor for tier setup.

## Problem Statement

### Who is affected

Merchants who use the Beco-style pricing tier pills feature on their full-page bundle widget. These are typically merchants running two to four bundle variants at different price points (e.g., "Buy 3 @ ₹499", "Buy 5 @ ₹799") and want shoppers to be able to switch between them on the same page.

### Pain points

1. **Theme Editor friction.** To configure tiers, merchants must navigate to `Online Store → Themes → Customize → Pages → [bundle page] → App Blocks → Full-Page Bundle Widget`. For non-technical merchants, finding the correct block within a nested theme customizer is error-prone and unintuitive.

2. **Bundle ID lookup required.** The bundle ID is an opaque CUID string (e.g., `clxzy12ab0000def`). Merchants must copy it from the admin app, switch to the Theme Editor, and paste it — introducing copy-paste errors.

3. **No validation.** The Theme Editor has no awareness of which bundle IDs are valid for the current shop. A merchant can enter any string, causing silent widget failures.

4. **Scale bottleneck.** As a merchant adds more bundle pages (each with their own tier set), maintaining tier config per-page via the Theme Editor becomes unmanageable.

5. **Disconnected from admin workflow.** The configure page in the admin app is already the canonical place where a merchant sets up a bundle. Tier configuration is logically part of that workflow — it is currently split across two separate surfaces.

## Research Findings

### How the current system works (codebase findings)

- The Liquid block `extensions/bundle-builder/blocks/bundle-full-page.liquid` reads up to 4 pairs of `tier_N_label` / `tier_N_bundle_id` from `block.settings`, serializes them into a JSON array, and emits `data-tier-config="[...]"` on the container `<div>`.
- The widget JS (`app/assets/bundle-widget-full-page.js`) reads `dataset.tierConfig`, parses it via `parseTierConfig()`, and calls `initTierPills()` to render pill buttons. Each pill carries `data-bundle-id` and switches the active bundle by re-fetching from the app proxy endpoint `GET /apps/product-bundles/api/bundle/:bundleId.json`.
- The primary bundle for the page is resolved via `page.metafields.custom.bundle_id` (set at "Place Widget Now" time), falling back through the page handle, legacy `$app` metafield, and finally the block setting.
- The `Bundle` model in Prisma has no tier-related fields. `BundlePricing` has a `rules` JSON field but it is for discount configuration, not tier linking.
- The existing page metafield strategy writes `custom:bundle_id` and `$app:bundle_id` via `metafieldsSet` at page-creation time inside `widget-full-page-bundle.server.ts`.

### Approach A — Store tiers in a new Prisma model (`BundleTier`), expose via app proxy

A new `BundleTier` model with `bundleId`, `tierOrder`, `label`, and `linkedBundleId` fields. The widget fetches tiers from a new app proxy endpoint at init time.

**Pros:**
- Fully normalized, queryable relational data.
- Each tier is independently addressable (add/remove/reorder without JSON marshalling).
- Can add metadata per tier in the future (e.g., tier description, image).

**Cons:**
- Requires a Prisma migration and new DB table.
- The widget needs an additional network request on init to fetch tier config (adds latency before pills are shown).
- More infrastructure complexity for what is essentially a small amount of config per bundle.

**Verdict:** Not recommended for v1. The additional network hop and migration cost are not justified when tier config is small and static at page-load time.

### Approach B — Store tiers as JSON in a metafield on the Shopify Page object; Liquid reads the metafield

A new `custom:tier_config` PAGE metafield (type `json`) is set whenever the merchant saves tiers in the admin. The Liquid block reads it via `page.metafields.custom.tier_config` and passes it to `data-tier-config`.

**Pros:**
- No new DB schema.
- Liquid reads it natively — no extra runtime API call.
- Consistent with the existing `custom:bundle_id` metafield pattern already in use.

**Cons:**
- Requires an additional Shopify API call (metafieldsSet mutation) on every tier config save.
- JSON metafields on pages are accessible in Liquid but require a metafield definition to be pre-created.
- The Liquid block must be updated to read from the metafield instead of (or in addition to) block settings.
- The page `shopifyPageId` must be known at save time — it is already stored on the `Bundle` record.

**Verdict:** Feasible. The `custom:bundle_id` precedent de-risks the metafield-on-page pattern. However, it disperses bundle configuration between the DB and Shopify's metafield store, making the admin app's DB the source of truth for some fields and Shopify for others. Syncing is required on every save, and the metafield may drift if the page is deleted and recreated.

### Approach C — Store tiers as JSON in the existing `Bundle` DB record; expose via existing bundle API

A new `tierConfig` JSON field is added to the `Bundle` model. The admin UI writes to this field. The existing `/apps/product-bundles/api/bundle/:bundleId.json` endpoint already returns the full bundle from the DB — `tierConfig` is included in the response. The Liquid block fetches tier config from the API response at widget init time, or the tier config is baked into the widget's initial `data-tier-config` by fetching it during Liquid render via the app proxy.

**Pros:**
- No new model. One `ALTER TABLE` migration adding a single nullable JSON column.
- Tier config lives alongside all other bundle config in the same DB row — single source of truth.
- No additional Shopify API call on save. Only the DB write.
- The app proxy endpoint already includes full bundle data — tier config is just another field.
- The widget already calls the bundle API to load bundle data; tier config can be included in that response.

**Cons:**
- The widget init currently renders tier pills synchronously from `data-tier-config` (before the async API call), so there is a brief render without pills if tier config comes only from the API.
- Tier config on Bundle A lists bundle IDs for B, C, D — the Liquid block still needs to know which bundles form a tier set for the current page. Since each page maps to one primary bundle, the tier config stored on that bundle covers its complete tier set.

**Verdict:** Recommended. Minimal schema change, no extra Shopify API dependency, consistent with all other bundle settings being stored in the DB.

### Approach D — Store in Bundle DB record, sync to Shopify metafield on the page at save time

Hybrid of B and C: write to `Bundle.tierConfig` in the DB AND call `metafieldsSet` to push `custom:tier_config` onto the Shopify Page object every time the merchant saves tiers.

**Pros:**
- Liquid can read tier config at render time (no widget-init fetch delay for pills).
- DB remains the source of truth; metafield is a derived cache.

**Cons:**
- Two writes on every save (DB + Shopify API). If the Shopify write fails, the metafield drifts from the DB.
- More code paths to maintain.
- The existing pattern (pills shown synchronously from `data-tier-config`) could be preserved with approach C by passing the tier config through the existing bundle API response and initializing pills after the bundle fetch — a minor UX trade-off.

**Verdict:** Rejected as over-engineered for v1. Prefer C.

## Recommended Approach: Option C

Store `tierConfig` as a JSON field on the `Bundle` model. The admin configure page provides the UI to build the tier set (select bundles, set labels). On save, `tierConfig` is persisted to the DB. The existing bundle proxy API (`/api/bundle/:bundleId.json`) includes `tierConfig` in its response. The widget reads tier config from the bundle API response and initializes pills after fetching bundle data (instead of from `data-tier-config` at HTML parse time).

The Liquid block's `data-tier-config` attribute is retained for backward compatibility with merchants who previously configured tiers via the Theme Editor — the widget will prefer API-sourced config if present.

## Functional Requirements

- **FR-01:** The full-page bundle configure page (`app.bundles.full-page-bundle.configure.$bundleId`) must have a "Pricing Tiers" section.
- **FR-02:** The merchant must be able to add 2–4 tiers. Each tier consists of a text label and a linked bundle (selected from existing bundles for that shop).
- **FR-03:** The bundle selector for each tier must show existing bundle names (not raw IDs).
- **FR-04:** Tier config must be saved to the `Bundle.tierConfig` JSON column when the merchant saves the configure page.
- **FR-05:** The existing `/api/bundle/:bundleId.json` proxy endpoint must include `tierConfig` in its response.
- **FR-06:** The widget JS must read `tierConfig` from the bundle API response and initialize tier pills from that data (in addition to the existing `data-tier-config` path for backward compat).
- **FR-07:** The Liquid block must not require any manual changes by the merchant for tier config to be applied when configured via the admin.

## Non-Functional Requirements

- **NFR-01 — Backward compatibility:** Existing bundles with no `tierConfig` in the DB must behave identically to today. Merchants who configured tiers via the Theme Editor block settings must still see their tiers (the widget still reads `data-tier-config` as a fallback).
- **NFR-02 — Performance:** Adding `tierConfig` to the bundle API response adds negligible payload. The widget already initiates an async fetch for bundle data — tier pills may render slightly after page load (~200–500 ms), which is acceptable given the pills are non-blocking UI.
- **NFR-03 — Data integrity:** `tierConfig` must be validated before DB write — each entry must have a non-empty label string and a valid bundle ID belonging to the same shop.
- **NFR-04 — No theme deploy required:** The feature must work with the existing deployed Liquid block and widget JS. Any changes to Liquid or widget JS require a `shopify app deploy`.

## User Stories (high level)

- As a merchant, I want to configure pricing tier pills from the bundle admin page so that I never have to open the Shopify Theme Editor.
- As a merchant, I want to pick linked bundles by name (not by ID) so that I don't need to copy-paste opaque IDs.
- As a merchant, I want the tier config I set in the admin to automatically appear on my storefront so that updates are reflected immediately without Theme Editor changes.
- As a developer, I want tier config stored in the DB alongside all other bundle fields so that there is a single source of truth for bundle configuration.

## Scope

### In Scope

- New `tierConfig` JSON column on `Bundle` model (Prisma migration).
- New "Pricing Tiers" UI section on the full-page bundle configure route.
- Bundle selector component (fetches the shop's active/draft full-page bundles).
- Save handler writing `tierConfig` to DB.
- Update to `/api/bundle/:bundleId.json` to include `tierConfig` in the response.
- Widget JS update to read `tierConfig` from API response and initialize pills from that data.
- Backward compatibility: `data-tier-config` fallback preserved.

### Out of Scope

- Syncing tier config to a Shopify page metafield (explicitly deferred — not needed for v1).
- Tier analytics or per-tier reporting.
- More than 4 tiers.
- Reordering tiers via drag-and-drop (labels and bundle selection are sufficient for v1).
- Tier config for product-page bundles (not applicable — tier pills are a full-page widget feature only).

## Risks

| Risk | Likelihood | Mitigation |
|------|------------|------------|
| Widget shows pills with a brief delay (renders after API response) | Medium | Acceptable UX trade-off; spinner/loading state covers the init period |
| Merchant's `tierConfig` references a bundle that has been deleted | Low | Validate bundle IDs at save time; widget `parseTierConfig` already filters invalid entries |
| Prisma migration on production DB causes downtime | Low | Single `ALTER TABLE ADD COLUMN` with nullable default — zero downtime in PostgreSQL |
| Merchant previously used Theme Editor tiers; admin tiers override | Low | Document priority order: API-sourced > `data-tier-config`. Old Theme Editor config continues to work if no admin config is set |
