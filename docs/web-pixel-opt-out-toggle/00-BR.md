# Business Requirement: Web Pixel Opt-Out Toggle

## Executive Summary
Merchants need the ability to disable the UTM attribution web pixel from their app dashboard, giving them control over whether customer event data is collected on their storefront.

## Problem Statement
The Wolfpack UTM Attribution Pixel is currently always active once the app is installed — there is no merchant-facing control to disable it. Merchants who do not use UTM tracking, have privacy concerns, or want to reduce storefront JavaScript execution have no way to opt out. This lack of control may deter privacy-conscious merchants from using the app.

## Research Findings

### Approach A: Toggle via Shopify Metafield (shop-level)
- **Pros:** No DB migration; metafields are native to Shopify; accessible from Liquid and APIs
- **Cons:** Metafields are a loose key-value store; querying across shops is harder; adds another metafield to manage

### Approach B: Toggle via App Database (new column on Shop model or new AppSettings model)
- **Pros:** Centralized with existing app data; easy to query; co-located with other settings
- **Cons:** Requires a DB migration; one more table/column to maintain

### Approach C: Delete/recreate the web pixel via GraphQL
- **Pros:** Pixel literally does not exist when disabled — zero overhead on storefront
- **Cons:** Slower toggle (API call to create/delete); potential race conditions with auto-activate logic on dashboard load

- **Recommended:** **Approach C (delete/recreate) with Approach B (DB flag) as the source of truth.** The DB stores the merchant's preference (`pixelEnabled`). When disabled, the pixel is deleted via `webPixelDelete` mutation. When re-enabled, it is recreated. The existing auto-activate logic on dashboard load and afterAuth is gated by this flag. This is the cleanest approach: when off, no pixel JS runs on the storefront at all.

## Functional Requirements (FRs)
- **FR-01:** A toggle control is visible on the app dashboard (or a dedicated settings area) allowing the merchant to enable/disable the web pixel.
- **FR-02:** When toggled OFF, the app deletes the web pixel from the store via the `webPixelDelete` GraphQL mutation.
- **FR-03:** When toggled ON, the app creates/updates the web pixel via `webPixelCreate` / `webPixelUpdate` GraphQL mutation.
- **FR-04:** The auto-activate logic (afterAuth + dashboard loader) respects the stored preference — it must NOT recreate a pixel the merchant has opted out of.
- **FR-05:** The toggle state persists across sessions (stored in the app database).
- **FR-06:** The attribution dashboard page should show a notice when the pixel is disabled.

## Non-Functional Requirements (NFRs)
- **NFR-01:** Toggling must complete within 3 seconds (single GraphQL mutation).
- **NFR-02:** Backward compatibility — existing shops without a stored preference default to pixel ENABLED (current behavior preserved).
- **NFR-03:** No data loss — disabling the pixel does not delete existing OrderAttribution records.

## User Stories (high level)
- As a **merchant**, I want to disable the UTM tracking pixel so that no customer event data is collected on my storefront.
- As a **merchant**, I want to re-enable the pixel after disabling it so that I can resume attribution tracking.
- As a **merchant**, I want a clear indicator of whether the pixel is currently active or not.

## Scope

### In Scope
- Toggle UI on the dashboard
- DB persistence of toggle state
- Pixel creation/deletion via Shopify GraphQL
- Gating existing auto-activate logic
- Notice on attribution dashboard when pixel is off

### Out of Scope
- Granular control over which events the pixel tracks (all-or-nothing toggle)
- Customer-facing privacy consent UI (Shopify's native privacy API handles this)
- Deleting existing OrderAttribution data when pixel is disabled

## Risks
| Risk | Likelihood | Mitigation |
|------|-----------|------------|
| Auto-activate logic recreates pixel after merchant disables it | High (if not gated) | Gate all auto-activate paths with DB flag check |
| Pixel creation fails silently on re-enable | Low | Return error to UI, show toast |
| Merchant confusion about Shopify privacy API vs. app toggle | Medium | Add helper text explaining what the toggle does |
