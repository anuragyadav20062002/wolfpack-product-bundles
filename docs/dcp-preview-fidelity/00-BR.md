# Business Requirement: DCP Preview Fidelity

## Executive Summary

The Design Control Panel (DCP) preview pane renders approximate, hand-coded React
approximations of bundle widget components. These approximations have drifted substantially
from the real storefront widget: wrong DOM structure, fake CSS gradients, no CSS variable
coupling, and dashed highlight boxes that are clipped by parent overflow. Merchants cannot
trust what they see in the preview.

## Problem Statement

When a merchant opens the DCP in the Shopify admin and customises a design setting (e.g.
button colour, card border radius, step bar style), the preview pane is supposed to show
them what the widget will look like on their storefront. Today it shows a rough sketch:

- The product card HTML bears no resemblance to the real widget's `.product-card` DOM.
- The step bar tabs are a plain horizontal strip; the real widget has pill-shaped `.step-tab`
  buttons with number badges, product thumbnails, checkmarks, and lock icons.
- The modal preview is a 320px box with no header, no close button, no grid, no footer.
- The promo banner shows text only — no image, no crop region.
- Dashed highlight boxes (`outline + outlineOffset`) are clipped when the highlighted
  element sits inside an `overflow: hidden` ancestor.
- **Zero** of the 100+ `--bundle-*` CSS custom properties are injected into the preview.
  Every setting is applied by hand-crafting an inline style that happens to match the
  variable's default value, so any non-default configuration shows wrong colours/sizes.

## Research Findings

### Current Preview Architecture

```
PreviewPanel.tsx
  → activeSubSection string routes to one of 8 sub-previews:
      ProductCardPreview, StepBarPreview, BundleHeaderPreview,
      BundleFooterPreview, PromoBannerPreview, GeneralPreview,
      GlobalColorsPreview (+ PromoTextPreview variant)

Each sub-preview:
  - Pure React JSX with inline `style={{}}` on every element
  - No real widget CSS loaded
  - Hardcodes colours, sizes, gradients as literal fallback values
  - Wires ~5–10 settings at most; ignores the remaining CSS variables
```

### Real Widget Architecture

```
ComponentGenerator.js (static class)
  → renderProductCard()     ← produces .product-card HTML
  → renderEmptyStateCards() ← produces .empty-state-card HTML
  → createModalHTML()       ← produces modal-overlay + modal-content HTML
  → renderTab()             ← produces .bundle-header-tab HTML
  → renderProgressBar()     ← produces .progress-bar HTML

bundle-widget-full-page.css (2,999 lines)
  → 100+ --bundle-* CSS custom properties
  → Variables cover: step timeline, tabs, product card, modal,
    footer, quantity selector, promo banner, global colours, fonts
```

### Gap Inventory

| Component | Preview shows | Real widget has | Severity |
|-----------|---------------|-----------------|----------|
| Product card | Custom React div, linear-gradient button | `.product-card > .product-image + .product-content-wrapper > .product-title + .product-price-row + .product-spacer + [variant-selector] + [inline-quantity-controls | product-add-btn]` | 🔴 High |
| Product card button | `linear-gradient(135deg, btn 0%, rgba(0,0,0,0.9) 100%)` (fake) | `background: var(--bundle-button-bg-color)` (flat) | 🔴 High |
| Inline qty controls | Uses `buttonBgColor` | Uses `--bundle-quantity-selector-bg-color` | 🟡 Medium |
| Step bar circles | 67px hardcoded | 69px via `--bundle-step-timeline-circle-size` | 🟢 Low |
| Step bar progress line | `top: 33px` hardcoded | `top: calc(var(--bundle-step-timeline-circle-size,69px)/2)` | 🟢 Low |
| Step bar tabs | Plain horizontal strip, no individual tabs | `.step-tab` pills with `.tab-number`, `.tab-images` (product thumbnails), `.tab-check`, `.tab-lock` | 🔴 High |
| Modal | 320px empty box | `modal-content > modal-header + modal-body(.product-grid) + modal-footer` with total pill, BACK/NEXT buttons, progress bar | 🔴 High |
| Dashed highlight boxes | `outline + outlineOffset` — clipped by `overflow:hidden` | Should not be clipped | 🔴 High |
| Promo banner | Coloured div with text only | Real image with computed `backgroundSize` / `backgroundPosition` from crop data | 🟡 Medium |
| CSS variable coverage | 0 of 100+ vars injected | 100+ `--bundle-*` vars drive all visual properties | 🔴 Critical |

### Approaches Considered

#### Option A: Incremental polish (keep inline-style architecture)
- Wire missing inline style properties to correct settings
- Fix product card structure by hand
- Fix tabs structure by hand
- Fix highlight box clipping (switch to `box-shadow` or wrapper overlay)
- Accept permanent maintenance burden: every widget DOM change requires a parallel preview change
- **Verdict:** ❌ Rejected — produces a permanently diverging copy; doesn't solve the root cause

#### Option B: Real CSS injection + ComponentGenerator HTML (Recommended)
- Import the real `bundle-widget-full-page.css` as a raw string (Vite `?raw` import)
- Scope all selectors by prefixing with `.dcp-preview-scope ` (one-time regex transform at import time)
- Inject the scoped CSS via a `<style>` tag inside the preview container
- Set all CSS variables as inline `style` on the `.dcp-preview-scope` wrapper div (computed from current `DesignSettings`)
- Render real `ComponentGenerator` HTML via `dangerouslySetInnerHTML`
- Result: the preview IS the real widget — colours, sizes, DOM structure and all

#### Option C: iframe isolation
- Full isolation but adds cross-frame messaging complexity, no shared React state
- **Verdict:** ❌ Rejected — engineering overhead not justified when Option B works cleanly

## Functional Requirements

- FR-01: Product card preview renders the real `.product-card` DOM structure and reflects the real CSS variables
- FR-02: Step bar tabs preview renders real `.step-tab` DOM structure with number badge, product thumbnails, and correct CSS variables
- FR-03: Step bar timeline circles and connecting line are positioned correctly and sized by CSS variable
- FR-04: Modal preview renders the real `modal-content` DOM with header, product grid, and footer
- FR-05: Dashed highlight boxes are never clipped — visible even when the highlighted element is inside `overflow: hidden`
- FR-06: All 100+ `--bundle-*` CSS variables are derived from `DesignSettings` and injected into the preview container
- FR-07: Promo banner preview renders with the configured image URL and crop region (if set)
- FR-08: The scoped CSS does not bleed into Polaris admin styles

## Non-Functional Requirements

- NFR-01: No external dependencies added
- NFR-02: Backward-compatible — existing `DesignSettings` type unchanged
- NFR-03: Preview re-renders within 100ms of a setting change
- NFR-04: Scoped CSS approach must not break if widget CSS is updated (CSS is imported at build time)

## Scope

### In Scope
- ProductCardPreview: rebuild with real ComponentGenerator HTML + scoped CSS
- StepBarPreview: rebuild tabs with real `.step-tab` HTML + scoped CSS
- BundleHeaderPreview: wire header CSS variables
- BundleFooterPreview: minor fixes only (structure is already reasonable)
- GeneralPreview: EmptyState uses real `.empty-state-card` HTML
- Dashed highlight box fix across all previews
- Settings → CSS variable mapping function (`settingsToCSSVars`)

### Out of Scope
- Animated interactions (product add, step progression)
- Real product data in preview (placeholder data is acceptable)
- Product-page widget preview (full-page is priority)
- Pixel-perfect mobile responsive preview
