# Business Requirement: Sync Bundle ŌĆö Stale Block Cleanup

## Executive Summary

Stores that installed the app before the migration from section blocks to app embed blocks have dead block references in their Shopify theme templates. These stale blocks render as invisible `display: none` wrappers that interfere with the app embed, causing bundles to appear blank on the storefront. Sync Bundle should clean up these stale references so merchants don't need to manually edit their theme.

## Problem Statement

**Who is affected:** Any merchant whose store was set up with the old section block approach (`bundle-full-page.liquid` for FPB, or the equivalent for PDP). When the app migrated to app embed blocks (`bundle-full-page-embed.liquid`), the old block files were removed from the extension. However, Shopify retains block references in theme template JSON files indefinitely ŌĆö even after the source Liquid file is deleted.

**What happens:**
1. Shopify renders stale block references as `<div class="shopify-block shopify-app-block" style="display: none;">` ŌĆö an empty hidden wrapper.
2. The app embed block also renders on the page (correctly), but the widget's DOM element (`#bundle-builder-app`) can end up inside the stale block's hidden wrapper instead of the embed's wrapper, depending on DOM order and the move script.
3. Result: the entire bundle widget is invisible. The widget JS initializes, product data loads, but everything is hidden behind `display: none`.

**How it was discovered:** On `wolfpackdemostore.myshopify.com/pages/toy-bundle`, the bundle appeared blank. DevTools investigation revealed the `shopify-block` wrapper had inline `display: none` ŌĆö caused by a dead `bundle-full-page` block reference in the page's custom template (`page.full-page-bundle.json`).

**Why current Sync doesn't fix it:** Sync Bundle already deletes and recreates the Shopify page (FPB) or product (PDP). However, it assigns the recreated page/product back to the same custom template suffix (`full-page-bundle` / `product-page-bundle`). That template still contains the stale block reference, so the problem persists.

## Research Findings

### Approach A: Remove the custom templateSuffix ŌĆö use default templates

- **Description:** Stop setting `templateSuffix: 'full-page-bundle'` on pages and `templateSuffix: 'product-page-bundle'` on products. New pages/products use the default `page.json` / `product.json` templates, which are clean.
- **Pros:**
  - Simple code change (remove templateSuffix from pageCreate/productCreate mutations)
  - No Theme API / Asset API needed
  - Immediately fixes the problem for synced bundles
  - App embed blocks already target all page/product templates via `enabled_on`, so no functionality is lost
- **Cons:**
  - Bundle pages/products share the default template with non-bundle pages/products ŌĆö merchants can't customize the template layout specifically for bundles
  - Existing pages already assigned to the custom template need to be migrated (set templateSuffix to null during sync)
  - If a merchant has intentionally customized `page.full-page-bundle.json` with additional sections (e.g., testimonials below the bundle), those customizations are lost

### Approach B: Clean stale blocks from the custom template via Theme Asset API

- **Description:** During sync, read the custom template JSON (`page.full-page-bundle.json`), find and remove stale block references (blocks with handles matching old removed blocks), and write the cleaned template back.
- **Pros:**
  - Preserves merchant customizations on the custom template
  - Surgically fixes only the stale blocks
- **Cons:**
  - Requires `read_themes` + `write_themes` scopes (or the newer `themeFiles` API)
  - The `themeFilesUpsert` API requires a Shopify exemption that disqualifies the app from the "Built For Shopify" badge (noted in existing code comments at `widget-full-page-bundle.server.ts:57`)
  - Higher risk ŌĆö writing malformed JSON to a template can break the merchant's page
  - More complex implementation

### Approach C: Remove templateSuffix + recreate page/product during sync

- **Description:** During sync, delete the existing page/product, recreate it with **no templateSuffix** (uses default template), and rewrite all metafields. This is a refinement of Approach A that explicitly clears the template assignment.
- **Pros:**
  - Same simplicity as Approach A
  - Sync already does delete+recreate ŌĆö just need to drop the templateSuffix
  - No Theme API needed
  - Guarantees a clean template on every sync
- **Cons:**
  - Same as Approach A ŌĆö loses custom template customizations
  - Merchants who add custom sections to their bundle page template will lose them on sync (mitigated by the fact that app embed + metafield cache handles all bundle rendering ŌĆö custom sections are rare)

### **Recommended: Approach C**

Approach C is the safest and simplest. The app embed block handles all rendering ŌĆö custom templates were originally needed for the section block approach but are unnecessary now. Dropping the templateSuffix during sync clears the stale blocks without needing Theme API access. The risk of losing merchant template customizations is low since the app embed renders independently of template structure.

## Functional Requirements (FRs)

- **FR-01:** During FPB Sync Bundle, the recreated Shopify page MUST NOT have a `templateSuffix` set (uses default `page.json` template).
- **FR-02:** During PDP Sync Bundle, the recreated Shopify product MUST NOT have a `templateSuffix` set (uses default `product.json` template).
- **FR-03:** During FPB "Place Widget Now", the created page MUST NOT have a `templateSuffix` set.
- **FR-04:** During PDP product creation (save/sync), the product MUST NOT have a `templateSuffix` set.
- **FR-05:** If a page/product already has a stale `templateSuffix`, Sync Bundle MUST clear it by setting `templateSuffix` to `null`/empty on the recreated entity.

## Non-Functional Requirements (NFRs)

- **NFR-01:** No new API scopes required ŌĆö must work within existing `write_content`, `write_products` scopes.
- **NFR-02:** No Theme API calls ŌĆö must not use `themeFilesUpsert` or Asset API.
- **NFR-03:** Backward compatible ŌĆö existing bundle data, metafields, and URLs must be preserved.
- **NFR-04:** Idempotent ŌĆö running Sync Bundle multiple times must produce the same result.

## User Stories (high level)

- As a merchant who installed the app before the embed migration, I want Sync Bundle to fix my broken bundle pages so that my storefront bundles render correctly without manual theme editing.
- As a new merchant, I want my bundle pages to work out of the box without needing to know about template suffixes or theme editor configuration.

## Scope

### In Scope
- Remove `templateSuffix` from FPB page creation (`createFullPageBundle`)
- Remove `templateSuffix` from PDP product creation/update
- Clear `templateSuffix` on existing pages/products during Sync Bundle
- Remove dead code related to template suffix management (ensureBundlePageTemplate, template checks)

### Out of Scope
- Cleaning stale blocks from theme template JSON files (Theme API approach ŌĆö rejected)
- Migrating all existing stores automatically (merchants click Sync Bundle themselves)
- Removing the old template files from themes (Shopify retains them; they become unused)

## Risks

| Risk | Likelihood | Mitigation |
|------|------------|------------|
| Merchant loses custom sections they added to `page.full-page-bundle.json` template | Low ŌĆö few merchants customize this template, and app embed handles all bundle rendering | Document in release notes; the old template still exists, merchant can re-assign it manually |
| Default `page.json` template has conflicting sections (e.g., rich text content area) that overlap with the bundle widget | Low ŌĆö the app embed hides the page title and the widget fills the content area | The widget's `hidePageTitle` setting + CSS handles this already |
| Shopify doesn't allow setting `templateSuffix` to null/empty on update | Very Low ŌĆö Shopify API supports `templateSuffix: ""` to clear it | Test on dev store before release |
