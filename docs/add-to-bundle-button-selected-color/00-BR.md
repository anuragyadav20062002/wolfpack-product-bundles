# Business Requirement: "Add to Bundle" Button Selected-State Color

## Executive Summary

Merchants cannot currently customise the background colour of the "Add to Bundle" / "Add to Box" button after a product has been selected. The selected state is hardcoded to `#10B981` (green) in the widget CSS, making it impossible to match individual brand palettes. This setting must be controllable from the Design Control Panel (DCP).

## Problem Statement

When a customer clicks "Add to Bundle", the button text changes to "Added to Bundle" and the background turns green (`#10B981` ŌĆö hardcoded). Merchants with non-green brand colours (e.g., purple, navy, red) cannot change this, breaking visual brand consistency on their storefront.

The unselected button background (`--bundle-button-bg`) is already DCP-configurable. The selected/added state is the only gap.

## Research Findings

**Current state in codebase:**
- Product-page widget: `.product-add-btn.added { background: #10B981; }` ŌĆö fully hardcoded, no CSS variable
- Full-page widget: Has `fullPageAddedButtonGradientStart` / `fullPageAddedButtonGradientEnd` fields ŌĆö these exist in `DesignSettings` and the CSS generator, but generate a gradient (2 stops). The user asked for a simple solid colour; gradient remains available as a secondary control.
- The CSS variable prefix convention is `--bundle-*`
- Default button colours differ by widget type (product page: `#FF9000`, full page: `#7132FF`)

**Approach A: Single solid colour field per widget type**
- Add `buttonAddedBgColor` (product page) and reuse/extend `fullPageAddedButtonGradientStart` (full page)
- Pros: Minimal surface area, one CSS var to add
- Cons: Full-page gradient fields already exist; adding a parallel solid colour could cause confusion

**Approach B: Unified solid colour + text colour fields, both widget types**
- Add two new `DesignSettings` fields: `buttonAddedBgColor` and `buttonAddedTextColor`
- Product page: `.product-add-btn.added` uses `--bundle-button-added-bg` and `--bundle-button-added-text`
- Full page: same CSS vars applied to its equivalent added-state element
- Pros: Clean, consistent, single DCP section, addresses both widget types uniformly
- Cons: Slightly more work (two fields vs one)

**Recommended: Approach B** ŌĆö The user specifically asked for both widget types and both bg + text colour is needed for proper contrast control (e.g., if bg turns dark, text must change too). The gradient fields on full-page are separate concerns (gradient accent within the button) and can coexist.

## Functional Requirements

- FR-01: Merchant can set the background colour of the "Add to Bundle" button **selected (added) state** via DCP
- FR-02: Merchant can set the text colour of the "Add to Bundle" button **selected (added) state** via DCP
- FR-03: Changes apply to **both** product-page and full-page bundle widgets
- FR-04: DCP preview shows the button in its added state with live colour updates
- FR-05: Default colours preserve current behaviour (`#10B981` background, `#FFFFFF` text)
- FR-06: Existing merchants with no saved value see the current green (`#10B981`) ŌĆö backward compatible

## Non-Functional Requirements

- NFR-01: **Zero breaking changes** ŌĆö default values match existing hardcoded colour
- NFR-02: No widget JS changes required ŌĆö purely CSS variable driven
- NFR-03: CSS variable follows existing `--bundle-*` naming convention
- NFR-04: Prisma nullable fields with DB defaults so existing rows remain valid without migration data seeding

## User Stories

- As a merchant, I want to change the "Added to Bundle" button background to my brand colour so that the selected state matches my storefront theme
- As a merchant, I want to change the "Added to Bundle" button text colour so that it remains readable against my chosen background
- As a merchant, I want to see a live preview of both selected and unselected button states in the DCP so I can compare before saving

## Scope

### In Scope
- `buttonAddedBgColor` + `buttonAddedTextColor` DCP settings
- CSS variable generation: `--bundle-button-added-bg`, `--bundle-button-added-text`
- `bundle-widget.css` update (`.product-add-btn.added` and full-page equivalent)
- DCP settings UI (in `ButtonSettings.tsx`)
- DCP preview update (in `ProductCardPreview.tsx`)
- Prisma schema + migration
- Default values matching current hardcoded colour

### Out of Scope
- Gradient added-button (the existing `fullPageAddedButtonGradientStart/End` fields already handle this separately for full-page)
- Animation / transition changes on the added state
- Hover state of the added button

## Risks

| Risk | Likelihood | Mitigation |
|------|------------|------------|
| Full-page widget uses different CSS selectors for added button | Medium | Verify in `bundle-widget.css` and `bundle-widget-full-page.js` before implementation |
| Existing Prisma rows missing the new columns | Low | Nullable columns with `@default` in schema |
| CSS specificity conflicts with existing hardcoded rule | Low | Replace hardcoded value with CSS var in same rule |
