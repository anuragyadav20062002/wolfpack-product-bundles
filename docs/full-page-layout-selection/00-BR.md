# Business Requirement: Full-Page Bundle Layout Selection

## Executive Summary
Allow merchants to choose between two full-page bundle UI layouts тАФ "Footer at Bottom" (current) and "Footer at Side" (new) тАФ during bundle creation, with a visual card-based selection modal that clearly previews each layout.

## Problem Statement
Currently, full-page bundles only support a single layout: a sticky footer pinned to the bottom of the viewport. A second layout option тАФ a side panel that sits to the right of the product grid тАФ provides a more desktop-friendly experience where the bundle summary, discount tiers, selected products, and navigation are always visible without scrolling. Merchants need the ability to choose which layout best fits their store's branding and UX.

## Research Findings

### Current Architecture
- **Bundle creation** starts on the dashboard (`app.dashboard/route.tsx`). A Polaris `Modal` opens with a name field, description field, and a two-card type selector (Product Page vs Full Page). The selected `bundleType` is sent as a hidden form field.
- **Database** uses a `BundleType` enum (`product_page | full_page`). There is **no** `layoutType` or `displayType` field.
- **Widget JS** (`bundle-widget-full-page.js`) reads `data-bundle-type="full_page"` from the Liquid template and calls `renderFullPageLayout()` тЖТ `renderFullPageFooter()` to build the bottom footer.
- **Footer CSS** uses `.full-page-footer.redesigned` with CSS custom properties for colors, typography, and spacing.
- **Liquid template** passes all config via `data-*` attributes on `#bundle-builder-app`.
- **DCP design settings** are stored per `BundleType` in the `DesignSettings` table and served as CSS variables via an app proxy endpoint.

### Approach A: New `layoutType` field on Bundle model
- Add `fullPageLayout` enum (`footer_bottom | footer_side`) to the `Bundle` Prisma model.
- Persist the layout choice in DB at creation time.
- Pass it to the widget via the API response (`/api/bundle/{id}.json`).
- Widget JS reads `bundle.fullPageLayout` and branches rendering accordingly.
- **Pros:** Clean DB-level separation, queryable, explicit per-bundle setting.
- **Cons:** Requires a Prisma migration.

### Approach B: Store layout type in bundle metadata/config JSON
- Store the layout choice inside an existing JSON field (e.g., pricing config or a new `config` JSON column).
- **Pros:** No migration needed.
- **Cons:** Not queryable, fragile, pollutes unrelated data structures.

### Approach C: Separate Liquid blocks per layout
- Create two Liquid blocks (`bundle-full-page-bottom.liquid`, `bundle-full-page-side.liquid`) with separate CSS/JS bundles.
- **Pros:** Clean separation at the extension level.
- **Cons:** Massive code duplication, violates DRY, doubles maintenance cost, complicates DCP.

### **Recommended: Approach A** тАФ New `fullPageLayout` enum field
Rationale: A dedicated DB field is the cleanest, most queryable, and most maintainable approach. A single Prisma migration is trivial. The widget JS already branches on `bundleType`; adding a `fullPageLayout` branch within the full-page code path is natural. The Liquid template already passes config via `data-*` attributes, so a new `data-full-page-layout` attribute is straightforward.

## Functional Requirements

- **FR-01:** When a merchant selects "Full Page" bundle type in the create modal, a second step appears asking them to choose a layout: "Footer at Bottom" or "Footer at Side".
- **FR-02:** The layout selection UI shows visual preview cards that clearly illustrate each layout.
- **FR-03:** The chosen layout is persisted on the `Bundle` record in the database.
- **FR-04:** The layout choice is exposed to the storefront widget via the bundle JSON API and Liquid `data-*` attribute.
- **FR-05:** The storefront widget renders the appropriate footer layout (bottom or side) based on the persisted setting.
- **FR-06:** Existing full-page bundles default to `footer_bottom` (backward compatible).
- **FR-07:** The layout choice is editable from the configure page (settings panel or a dedicated control).

## Non-Functional Requirements

- **NFR-01:** Zero downtime migration тАФ existing bundles must continue working with no manual intervention.
- **NFR-02:** DRY тАФ the side-footer and bottom-footer must share as much rendering code as possible (product tiles, discount messaging, navigation buttons, pricing calculations).
- **NFR-03:** The layout selection modal must be visually polished and consistent with the existing Polaris-based admin UI.
- **NFR-04:** The widget CSS for the side layout must use the same CSS custom property system as the bottom footer for DCP compatibility.

## User Stories (High Level)

1. As a **merchant**, I want to choose between a bottom-footer and side-footer layout when creating a full-page bundle, so that I can pick the design that best fits my store.
2. As a **merchant**, I want to see a visual preview of each layout option before committing, so I know what I'm selecting.
3. As a **customer**, I want the side-footer layout to show my bundle summary, selected products, discount tiers, and navigation in a persistent sidebar, so I can track my progress without scrolling.
4. As a **merchant**, I want to change the layout of an existing full-page bundle from the configure page.

## Scope

### In Scope
- Layout selection step in the bundle creation modal (dashboard)
- Visual preview cards for both layouts
- `fullPageLayout` field on `Bundle` model (Prisma migration)
- API endpoint update to include `fullPageLayout` in response
- Liquid template update to pass `data-full-page-layout`
- Widget JS: side-footer rendering (new `renderSideFooter()` method sharing components with `renderFullPageFooter()`)
- Widget CSS: side-footer styles (`.full-page-footer.side-layout`)
- Configure page: layout display/edit control
- Backward compatibility: existing bundles default to `footer_bottom`

### Out of Scope
- Product-page bundle layout variations (out of scope)
- Mobile-specific side-footer behavior (side footer will stack below on mobile via CSS)
- DCP preview for side-footer (can be added later)
- Theme editor setting to override layout (widget reads from DB, not theme editor)

## Risks

| Risk | Likelihood | Mitigation |
|------|------------|------------|
| Prisma migration fails on existing data | Low | Default value `footer_bottom` ensures backward compat |
| Side footer breaks on narrow viewports | Medium | CSS media query collapses side footer to bottom on mobile |
| Merchants confused by two layout options | Low | Visual preview cards make the difference immediately clear |
| DCP design settings don't apply to side footer | Medium | Reuse same CSS variable system; add side-specific vars only where needed |
