# Business Requirement: Custom Brandable URL Slugs for Full-Page Bundles

## Executive Summary
Full-page bundle pages currently receive machine-generated, UUID-derived URLs (e.g., `/pages/bundle-6741c3e5f4b2a8c3d9e1f2a3`) that are unusable for marketing, SEO, or sharing. Merchants need the ability to define clean, brandable page slugs — just as every major competitor already supports.

---

## Problem Statement

**Who experiences it:** Every merchant who has placed a full-page bundle on their storefront.

**Pain points:**
1. **Unshare-able links** — A URL like `/pages/bundle-clxyabc123def456` cannot appear in an ad, email campaign, or social post without embarrassing the brand.
2. **Zero SEO value** — Search engines cannot crawl or attribute keyword intent to opaque ID-based paths. A URL like `/pages/build-your-own-skincare-kit` carries meaningful keyword signals.
3. **Customer confusion** — URL-bar copy reads as a technical artefact rather than a product destination.
4. **No merchant autonomy** — The handle is silently auto-assigned at placement time with no review or override opportunity.

**Competitor context:**
Beco BYOB (`/pages/BYOB_13May`), as well as every other theme-based Shopify bundle app, uses the standard Shopify Page + Theme App Extension pattern. Merchants control the page URL the same way they control any other Shopify page — by setting the handle. We are the only party in the flow that removes that control.

## Research Findings

### Current Implementation
- **Handle generation:** `bundle-${bundleId.toLowerCase().replace(/[^a-z0-9]+/g, '-')}` — hardcoded in `widget-full-page-bundle.server.ts:65`
- **Bundle ID resolution:** Liquid template reads `page.metafields.custom.bundle_id` (PRIMARY). Handle extraction is only a secondary fallback and can be safely retired.
- **`pageUpdate` mutation:** Already present in `widget-full-page-bundle.server.ts:180-189`; currently updates `templateSuffix` only — trivially extendable to update `handle`.
- **Prisma model:** `shopifyPageHandle String?` already stored and indexed on the `Bundle` model.
- **Form state:** `useBundleForm.ts` — no custom slug field yet; adding one is a pure extension.

### Approach A: Merchant-controlled slug, set in admin before placement
Merchant types their desired slug (e.g., `build-your-own-kit`) directly in the configure page. Backend uses this slug when calling `pageCreate`. On subsequent saves the handle is updated via `pageUpdate`.

- **Pros:** Maximum merchant control; clean UX; zero extra API calls; fully consistent with how Shopify itself works.
- **Cons:** Requires client-side slug validation + uniqueness check.
- **Verdict:** ✅ **Recommended**

### Approach B: Auto-generate slug from bundle name, no merchant editing
System slugifies the bundle name at placement time (`build-your-own-kit` from "Build Your Own Kit") — merchant cannot customise further.

- **Pros:** Zero friction, no new UI needed.
- **Cons:** Merchant still cannot align URL with campaign name; collisions on duplicate bundle names require silent suffix appending, leading to surprise handles.
- **Verdict:** ❌ Rejected — addresses 60 % of the problem while adding hidden fragility.

### Approach C: Dedicated "page management" sub-section
Full CRUD for Shopify pages inside the app.

- **Pros:** Most powerful.
- **Cons:** Massive scope; overlaps Shopify's native admin; out of proportion with the problem.
- **Verdict:** ❌ Out of scope.

**Recommended:** Approach A, with Approach B's name-based default pre-filled in the input so merchants can accept it in one click or customise it freely.

## Functional Requirements

- **FR-01** — A "Page URL" text field appears in the full-page bundle configure page.
- **FR-02** — The field auto-populates with a slugified version of the bundle name (e.g., "Build Your Own Kit" → `build-your-own-kit`) when the bundle has not yet been placed.
- **FR-03** — A live URL preview label reads `your-store.myshopify.com/pages/[slug]` and updates as the merchant types.
- **FR-04** — On "Add to Storefront", the system uses the merchant-supplied slug (or the auto-generated default if unchanged) as the Shopify page handle.
- **FR-05** — If the merchant edits the slug after placement and saves, the system renames the existing Shopify page via `pageUpdate`.
- **FR-06** — The slug field enforces Shopify handle rules: lowercase, alphanumeric + hyphens only; no leading/trailing hyphens; 1–255 characters.
- **FR-07** — If the chosen handle is already taken by another Shopify page in the same shop, the system appends a numeric suffix automatically (e.g., `build-your-own-kit-2`) and surfaces a non-blocking notice.
- **FR-08** — Existing bundles with auto-generated handles (the `bundle-{id}` pattern) continue to work without any migration; they get the new slug field pre-filled with their existing handle for easy editing.

---

## Non-Functional Requirements

- **NFR-01 Performance** — Slug uniqueness check must not add perceptible latency to the Save action; a background validation on blur is acceptable.
- **NFR-02 Backward compatibility** — Zero existing bundle pages break. Metafield-based bundle ID resolution (the primary mechanism) is unaffected by handle changes.
- **NFR-03 Idempotency** — Placing the widget twice with the same slug must be safe (re-uses the existing page, does not create a duplicate).
- **NFR-04 Data integrity** — `shopifyPageHandle` in the DB must always reflect the live Shopify page handle to keep "View on Storefront" links accurate.
- **NFR-05 Liquid fallback** — The secondary `bundle-` prefix extraction fallback in `bundle-full-page.liquid` should be gracefully retired (or left in place as dead code) since it will stop working for new slugs; metafields remain the authoritative source.

---

## User Stories (high level)

1. As a merchant, I want to set a custom URL slug for my bundle page so that I can share brandable links in marketing campaigns.
2. As a merchant, I want the system to suggest a sensible default slug based on my bundle name so that I don't have to think of one from scratch.
3. As a merchant, I want to rename a live bundle page's URL after it has been placed so that I can align it with a seasonal campaign without re-creating the bundle.
4. As a merchant, I want to see a live preview of the full URL before committing so that I can confirm it looks right.

---

## Scope

### In Scope
- New "Page URL" field in the full-page bundle configure page
- Slug validation (format + uniqueness)
- Name-based default generation
- Live URL preview
- Pass custom slug through to `createFullPageBundle()` / `pageUpdate`
- Update `shopifyPageHandle` in DB after every change
- Pre-fill field with existing handle for already-placed bundles

### Out of Scope
- Redirects from the old UUID-based URL (merchants can set these in Shopify admin if needed)
- Bulk slug migration for all existing bundles
- Product-page bundles (they don't have standalone pages)
- Custom domain mapping

## Risks

| Risk | Likelihood | Mitigation |
|------|-----------|------------|
| Merchant picks a handle already used by another page | Medium | Auto-suffix collision handling (FR-07) |
| Merchant changes slug mid-campaign, breaking inbound links | Low | No redirects auto-created — documented as expected behaviour; Shopify admin redirects can be used manually |
| `pageUpdate` handle change triggers Shopify 422 if handle taken | Low | Pre-validate via `pages` REST API or GraphQL query before mutation |
| Existing `bundle-{id}` fallback Liquid extraction becomes stale | Low | Metafield is already primary; fallback removal is cosmetic cleanup |
