# Business Requirement: Analytics Pixel Toggle

## Executive Summary

Merchants must explicitly opt in to UTM attribution tracking via a toggle on the Analytics page. The toggle reflects live pixel connection state and calls the existing pixel activation/deactivation API when changed.

## Problem Statement

**Who:** Merchants using the Wolfpack Product Bundles app who care about UTM attribution.

**Pain points:**
1. The pixel was previously activated automatically on install — merchants had no visibility or control over whether tracking was running.
2. There is no UI to tell the merchant whether the pixel is currently connected or disconnected.
3. Recovering from a disconnected pixel requires developer intervention (hitting `/api/activate-pixel` manually or reinstalling the app).
4. Merchants in jurisdictions with strict consent requirements (GDPR, CCPA) need explicit opt-in controls for any tracking pixel.

## Research Findings

The analytics page (`app/routes/app/app.attribution.tsx`) is a read-only dashboard — it has a loader but no action handler. The pixel activation logic already exists in `app/services/pixel-activation.server.ts` (`activateUtmPixel`) and `app/routes/api/api.activate-pixel.tsx`. A deactivation path (webPixelDelete) does not yet exist as a standalone service function.

- **Approach A: Dedicated settings page** — Put the pixel toggle on a separate "Settings" or "Integrations" page.
  - Pros: Clean separation of concerns; analytics page stays pure data.
  - Cons: Low discoverability — merchants won't find it naturally when they visit analytics.

- **Approach B: Toggle on the Analytics page** — Add a card at the top of the analytics page with the pixel status and enable/disable toggle.
  - Pros: Contextually correct — the merchant is already looking at attribution data when they manage the pixel. Highest discoverability.
  - Cons: Adds a non-data element to a data page (minor).

- **Recommended:** Approach B — Toggle on Analytics page. The analytics page is exactly where a merchant will notice "why am I seeing no data?" and will want to enable tracking. Keeps the fix one click away from the symptom.

## Functional Requirements

- FR-01: On page load, query Shopify's `webPixel` to determine whether the pixel is currently active for this shop.
- FR-02: Display the pixel status ("Active" / "Not Active") alongside a toggle control.
- FR-03: When the toggle is turned ON (pixel not currently active), call `activateUtmPixel()` and show a success or error toast.
- FR-04: When the toggle is turned OFF (pixel currently active), delete the web pixel and show a success or error toast.
- FR-05: The toggle must be disabled (loading state) while an activation or deactivation request is in-flight.
- FR-06: The pixel must NOT be activated automatically on install — the `afterAuth` hook must no longer call `activateUtmPixel()`.
- FR-07: If activation fails because the extension has not been deployed, the error toast must include a human-readable message.

## Non-Functional Requirements

- NFR-01: **No page reload** — the toggle state updates optimistically (or from server response) without a full navigation.
- NFR-02: **Backward compatibility** — shops that already had the pixel activated (from the old auto-activate logic) must see the toggle in the ON state on first load.
- NFR-03: **No new DB columns** — pixel state is source-of-truth from Shopify's API, not a local flag.

## User Stories (high level)

- As a merchant, I want to see whether UTM tracking is currently enabled for my store so I know if attribution data is being collected.
- As a merchant, I want to enable UTM tracking with one click on the analytics page so I don't need developer help.
- As a merchant, I want to disable UTM tracking so I can comply with privacy regulations or pause data collection.
- As a merchant, I want clear feedback (success/error toasts) when I change the tracking state so I know whether the action succeeded.

## Scope

### In Scope
- Pixel status query in the analytics page loader
- Enable/disable action handler on the analytics route
- Toggle UI on the analytics page (Polaris `SettingToggle` or `Card` + `Button`)
- Toast feedback for enable/disable outcomes
- Remove automatic pixel activation from `afterAuth`
- Add `deactivateUtmPixel()` service function alongside existing `activateUtmPixel()`

### Out of Scope
- Per-bundle or per-event granular tracking controls
- Consent banner integration
- Pixel status webhooks
- Any changes to the pixel extension code itself

## Risks

| Risk | Likelihood | Mitigation |
|------|------------|------------|
| Extension not deployed when merchant enables pixel | Medium | Show specific error: "Extension must be deployed first via Shopify CLI" |
| Shopify API latency causes toggle to appear stuck | Low | Disable toggle during in-flight request; re-enable on response |
| Merchant on older install never had pixel — toggle shows OFF correctly | Low | Loader treats missing pixel as OFF state — correct by design |
