# Business Requirement: DCP Federation — Separate Bundle Widget Customization Paths

## Executive Summary
Merchants currently experience confusion when customizing bundle widget styles because the Design Control Panel exposes settings for both the full-page bundle widget and the product-page bundle widget in a single undifferentiated interface. Federating these into separate, clearly labelled entry points will reduce errors and improve the customization experience.

## Problem Statement
The DCP modal presents a unified sidebar that mixes sections belonging exclusively to the full-page widget (Promo Banner, Pricing Tier Pills, Step Timeline) with sections that belong to the product-page widget (Modal Close Button, drawer settings) and sections shared by both. Merchants must mentally filter which settings apply to which widget, and there is no visual feedback indicating which widget a section affects. The result: merchants accidentally over-write settings for a widget they didn't intend to edit, or cannot find relevant settings due to the noise from irrelevant sections.

Additionally, the `selectedBundleType` state variable in `useDesignControlPanelState` defaults to `product_page` with no UI toggle — meaning all saves through the current modal always target `product_page`, silently leaving `full_page` settings untouched.

## Research Findings

### Approach A: Tab Toggle in Current Modal
Add a "Full Page / Product Page" tab switcher at the top of the existing modal. Pros: minimal surface-area change. Cons: still one modal, user must consciously switch; easy to miss; sidebar still renders all sections and must be filtered by JS — adds state complexity.
**Verdict: ❌ Rejected** — does not solve the root confusion, just adds a hidden toggle.

### Approach B: Separate Full-Screen Modals Per Bundle Type (Recommended)
Replace the single DCP page action button with a landing page containing two entry-point cards. Each card opens its own `<Modal variant="max">` pre-configured for that bundle type. The landing page also contains the Custom CSS section as a shared power-user tool. The NavigationSidebar rendered inside each modal will show only sections relevant to that bundle type.
**Verdict: ✅ Recommended** — natural separation of concerns; leverages existing modal infrastructure; no schema changes required; backward-compatible.

### Approach C: Separate Routes (Sub-pages)
Each bundle type gets its own Remix route (e.g. `/app/design-control-panel/full-page`). Pros: deep-linkable URLs. Cons: full-screen modals in Polaris App Bridge already provide a near-identical UX with far less routing overhead; would require duplicating loader/action logic.
**Verdict: ❌ Rejected** — unnecessary routing overhead for a modal-heavy UI.

## Functional Requirements

- FR-01: The DCP landing page MUST display two card-based entry points — one for "Landing Page Bundles" (full_page) and one for "Product Bundles" (product_page).
- FR-02: Each card MUST include an SVG illustration representing its widget type, a title, and a short description.
- FR-03: Clicking a card MUST open a dedicated full-screen modal pre-loaded with settings for that bundle type only.
- FR-04: The modal for each bundle type MUST display only the sections relevant to that type in the NavigationSidebar.
- FR-05: Saving inside a bundle-type modal MUST target only that bundle type's DB row.
- FR-06: The Custom CSS section MUST be separated from both modals and rendered on the landing page.
- FR-07: The Custom CSS section MUST support per-bundle-type CSS (separate `customCss` fields for each type).
- FR-08: A "View CSS Guide" button MUST open a guide modal listing commonly customized CSS classes and variables.
- FR-09: The preview rendered inside each modal MUST use the exact same CSS variables and structural CSS as the live storefront widgets.
- FR-10: All unsaved-change detection and SaveBar behavior MUST work independently per modal.

## Non-Functional Requirements

- NFR-01: **Backward Compatibility** — existing `DesignSettings` rows in the DB must remain valid; no schema changes.
- NFR-02: **Preview Fidelity** — preview components must use `PreviewScope` with the real CSS variable pipeline, not approximations.
- NFR-03: **Performance** — each modal should only compute CSS variables for its own bundle type; no cross-type recomputation.
- NFR-04: **Accessibility** — modal open/close must respect App Bridge focus trap and escape-key dismissal.

## User Stories (high level)

- As a merchant, I want to see clearly separated cards for each bundle widget type so I know exactly which widget I'm customizing.
- As a merchant, I want the customization modal to show only relevant settings for the selected widget type so I don't accidentally change settings that don't apply.
- As a merchant, I want a dedicated Custom CSS section with a guide so I can apply advanced overrides without navigating away from the page.
- As a developer, I want the preview in each modal to be pixel-accurate to the live storefront so I can be confident my changes look correct before saving.

## Scope

### In Scope
- Redesign DCP landing page with two bundle-type cards + SVG illustrations
- Create two separate `<Modal variant="max">` instances (one per bundle type)
- Filter NavigationSidebar sections per bundle type
- Per-bundle-type Custom CSS with shared guide modal
- Correct SaveBar wiring per modal

### Out of Scope
- Changes to the DesignSettings Prisma schema
- Changes to the CSS variable generation pipeline
- Changes to widget source JS/CSS
- Changes to the public CSS delivery API
- New settings sections or new CSS variables

## Risks

| Risk | Likelihood | Mitigation |
|------|------------|------------|
| Two modals sharing state causes cross-contamination | Low | Each modal gets its own copy of settings from the loader; state is keyed by bundleType |
| NavigationSidebar filtering breaks existing section keys | Low | Filter is additive — existing keys remain; just a subset is rendered per modal |
| Two SaveBar instances conflict | Low | Each modal has its own SaveBar ID; App Bridge scopes them to their modal |
