# Business Requirement: FPB Layout Selection at Creation Step

## Executive Summary
Move the full-page bundle footer layout selection (Floating Cart Card vs Sidebar Panel) from the FPB configure page sidebar into the bundle creation modal, so merchants commit to a layout before the bundle is built rather than discovering it after.

## Problem Statement
**Who:** Merchants creating full-page bundles.

**Pain point:** The layout choice (floating footer vs sidebar) fundamentally changes the visual structure of the bundle page — products grid + floating cart vs two-column layout with sidebar panel. Currently this decision lives in the left sidebar of the configure page, tucked after the merchant has already named their bundle and started configuring steps. Merchants either miss it entirely (defaulting to footer_bottom without realising) or have to reconfigure after switching layouts mid-build.

The creation step already captures the bundle name, description, and bundle type (Full Page vs Product Page). Layout is a foundational structural choice that belongs at creation time alongside bundle type — not discovered later.

## Research Findings

### Current Implementation
- **Layout picker location:** FPB configure page, `app/routes/app/app.bundles.full-page-bundle.configure.$bundleId/route.tsx` lines 1516–1623 — always-visible Card in the left sidebar
- **SVGs:** Two inline SVG illustrations (140×96 viewBox each), not extracted into components
- **Enum:** `FullPageLayout { FOOTER_BOTTOM = "footer_bottom", FOOTER_SIDE = "footer_side" }` — defined in `app/constants/bundle.ts:30`
- **DB default:** `fullPageLayout @default(footer_bottom)` — Prisma `schema.prisma:103`
- **Creation handler:** `handleCreateBundle` in `app/routes/app/app.dashboard/handlers/handlers.server.ts:397` already reads `fullPageLayout` from formData but it is never submitted (no UI in the modal)

### Approach A — Move SVG picker into the Create Bundle modal (Recommended)
Add a third "step" to the Create Bundle modal flow, shown only when Full Page bundle type is selected. The SVG cards and selection logic are lifted from the configure page and placed here. The `fullPageLayout` value is submitted with the creation form and saved at creation time.

**Pros:** Merchant makes the structural decision up-front; configure page becomes cleaner (remove the always-visible left-sidebar card); creation handler already accepts `fullPageLayout` in formData; zero schema changes needed.

**Cons:** Modal becomes slightly longer for FPB creation (acceptable — same pattern as existing type selection step).

### Approach B — Make layout a required field with inline prompt on configure page
Keep the picker on configure page but surface it as a required blocking step before the rest of the UI is shown.

**Cons:** Still post-creation friction; configure page already has a 4-tab nav structure that this would disrupt.

**Recommended:** Approach A — move to creation modal.

## Functional Requirements
- FR-01: The Create Bundle modal must show a layout selection step when "Full Page Bundle" type is selected.
- FR-02: The layout step must display both options with their existing SVG illustrations and labels.
- FR-03: The selected layout must be submitted with the creation form and persisted on the bundle record.
- FR-04: The layout selection Card must be removed from the FPB configure page left sidebar.
- FR-05: Existing FPB bundles with a saved `fullPageLayout` must continue to work unchanged (no migration needed — DB field already exists with correct values).
- FR-06: The layout step must default to `footer_bottom` if the user skips selection (preserves existing default).

## Non-Functional Requirements
- NFR-01: No Prisma schema changes — `fullPageLayout` field already exists.
- NFR-02: No widget JS changes — widget reads `fullPageLayout` from bundle config, unaffected by where the UI lives.
- NFR-03: The modal must remain accessible and mobile-responsive.
- NFR-04: No change to the configure page tabs/navigation structure beyond removing the layout Card.

## User Stories
- As a merchant creating a full-page bundle, I want to choose between a floating cart footer and a sidebar panel during creation, so I build the right layout from the start.
- As a merchant, I want the configure page to be focused on step and product configuration, not structural layout choices.

## Scope
### In Scope
- Layout picker UI moved into Create Bundle modal (FPB flow only)
- Remove layout picker from configure page left sidebar
- SVG illustrations moved with the picker

### Out of Scope
- Adding new layout options
- Changing how the widget renders layouts
- Changing the configure page tab structure
- Any Prisma migrations

## Risks
| Risk | Likelihood | Mitigation |
|------|------------|------------|
| Existing FPB bundles have `fullPageLayout` already set | Low — no risk | DB field exists, no migration needed |
| Modal UX becomes cluttered | Low | Layout step appears only for Full Page type selection |
