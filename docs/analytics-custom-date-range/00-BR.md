# Business Requirement: Analytics Custom Date Range

## Executive Summary

Merchants need the ability to select a custom date range on the analytics page, rather than being limited to fixed presets (7 / 30 / 90 days), so they can analyse revenue trends and UTM attribution for their own business cycles (promotions, launches, seasons).

## Problem Statement

The current analytics page (`/app/attribution`) offers only three fixed date presets via a `Select` dropdown. Merchants cannot inspect a specific sale window (e.g. "Black Friday 2025 — Cyber Monday 2025"), compare a custom promotion period, or align their reporting to their fiscal/business calendar. This reduces the analytics page's usefulness for any merchant with a non-standard reporting cadence.

## Research Findings

### Approach A: Polaris DatePicker in a Popover (calendar overlay)
- User clicks a trigger button showing the current range (e.g. "Mar 1 – Mar 25")
- A `Popover` opens containing a `DatePicker allowRange multiMonth` calendar
- User selects start and end dates; "Apply" button navigates with `?from=YYYY-MM-DD&to=YYYY-MM-DD`
- Pros: full visual date selection, familiar UX, supports any range
- Cons: more complex component, controlled `month`/`year` state required, first usage of Polaris `DatePicker` in the codebase
- **Recommended** ✅

### Approach B: Two `TextField type="date"` inputs (HTML date inputs)
- Two native HTML date pickers rendered inline
- Pros: zero Polaris complexity, native browser pickers
- Cons: inconsistent cross-browser rendering, clashes with Polaris design system, poor UX on Safari
- **Rejected** ❌

### Approach C: Keep the preset Select and add a "custom" option that reveals text fields
- Hybrid — `Select` gains a "Custom" option; when selected, two text inputs appear
- Pros: minimal UI change
- Cons: confusing two-step interaction, still has the native date picker UX problems
- **Rejected** ❌

### Recommended: Approach A
The Polaris `DatePicker` is already installed (v13+) with `allowRange` and `multiMonth` support. The `?from&to` URL param shape is clean and backwards-compatible with the existing `?days` preset routing. The `datePickerWrap` CSS class already exists in the stylesheet (was named in anticipation of this).

## Functional Requirements (FRs)

- FR-01: The date range selector on the analytics page must support arbitrary start/end date selection in addition to the three existing presets
- FR-02: Selecting a custom date range must navigate to `?from=YYYY-MM-DD&to=YYYY-MM-DD`
- FR-03: The loader must accept `from` + `to` params and compute `since`, `prevSince`, and `days` from them
- FR-04: The previous period must be a symmetric window of equal length immediately preceding the custom range
- FR-05: `buildBundleTrendSeries` must accept an explicit `until` date so weekly/daily bucketing is computed from actual span, not only from a `days` integer
- FR-06: The UI must correctly seed and display the active range from loader data (handles direct URL navigation)
- FR-07: Preset options (7 / 30 / 90 days) must continue to work unchanged

## Non-Functional Requirements (NFRs)

- NFR-01: No new DB queries — the loader continues to make exactly two Prisma queries (current + previous period)
- NFR-02: Zero regressions on preset date ranges
- NFR-03: The `disableDatesAfter={new Date()}` must be set to prevent future date selection
- NFR-04: The feature must be backwards-compatible — URLs with only `?days=N` continue to work

## User Stories (high level)

- As a merchant, I want to select a specific start and end date so I can analyse revenue for any custom period
- As a merchant, I want to see the previous period automatically computed so I can compare against the period immediately before my selection
- As a merchant, I want to bookmark or share a custom date range URL so I can return to the same view

## Scope

### In Scope
- Polaris `DatePicker` in a `Popover` trigger replacing the current `Select` (while keeping presets as quick-select chips/buttons)
- Loader changes to handle `from`/`to` URL params
- `buildBundleTrendSeries` update to accept explicit date range
- CSS additions for the popover/calendar panel

### Out of Scope
- Saving custom date range preferences per merchant
- Comparing two arbitrary non-contiguous ranges
- Export/download functionality

## Risks

| Risk | Likelihood | Mitigation |
|------|------------|------------|
| Polaris DatePicker `month`/`year` controlled state complexity | Medium | Use `useState` with `onMonthChange` callback; well-documented pattern |
| `buildBundleTrendSeries` API change breaks existing callers | Low | The function is only called once (in the loader); add `until` param with default = `new Date()` |
| Very large custom ranges (> 1 year) generating too many data points | Low | Cap the trend chart at weekly buckets for ranges ≥ 90 days; no change to DB query structure |
