# Business Requirement: Toast Notification â€” Full DCP Customization

## Executive Summary
Bundle widget toast notifications currently expose only two merchant-configurable properties (background color, text color), leaving eight visual properties hardcoded. Merchants need full control to match the toast to their store brand including shape, border, animation behavior, font, and shadow.

## Problem Statement
When merchants customize their bundle widget through the DCP, they can fine-tune almost every UI element â€” product cards, footer, header, buttons. The toast notification is the one component that resists branding: its border-radius (8px), box-shadow, animation speed (0.3s), font size (13px), font weight (500), and position (slides from top) are all hardcoded. Merchants who want pill-shaped toasts, soft shadows, slow fade-in animations, or bottom-entry behavior (as seen on the Skai Lama demo store) have no path other than custom CSS, which requires developer knowledge.

## Research Findings

### Approach A: Migrate all hardcoded values to CSS variables, JS reads one variable for position
For each hardcoded property in `.bundle-toast`, introduce a `--bundle-toast-*` CSS variable. The widget JS `ToastManager.show()` reads `--bundle-toast-enter-from-bottom` (1 or 0) and conditionally adds a `.bundle-toast-from-bottom` class, which CSS uses to switch the animation direction and vertical position.
- **Pros:** Pure CSS variable approach for all styling (consistent with rest of DCP); single JS read only for the directional toggle; no breaking changes to the ToastManager API; works in both widget types from a single CSS file.
- **Verdict: âś… Recommended**

### Approach B: Store all toast config in a new `toastSettings` JSON blob (like footerSettings)
Add a new `toastSettings Json?` column to DesignSettings and store all 8 new fields there.
- **Pros:** Consistent with how existing "section" settings are grouped.
- **Cons:** JSON blobs are acknowledged tech-debt in this codebase (arbitrary split between direct columns and blobs); existing toast fields (`toastBgColor`, `toastTextColor`) live in `generalSettings`, creating further inconsistency; direct columns are the newer preferred approach.
- **Verdict: âťŚ Rejected**

### Approach C: Extend the existing `generalSettings` JSON blob
Add the 8 new fields to `extractGeneralSettings` (handlers.server.ts).
- **Pros:** No schema migration needed; toast fields stay together.
- **Cons:** Compounds the existing tech-debt; mergeSettings spreads JSON blobs last, which means any new direct column with the same name would be silently overridden; harder to query specific fields.
- **Verdict: âťŚ Rejected**

**Recommended:** Approach A â€” 8 new direct Prisma columns + CSS variable pipeline + JS direction toggle.

## Functional Requirements
- FR-01: All 8 new toast properties MUST be configurable through the DCP Toasts settings panel.
- FR-02: `toastBorderRadius` MUST control `.bundle-toast { border-radius }`.
- FR-03: `toastBorderColor` + `toastBorderWidth` MUST add a visible border to the toast.
- FR-04: `toastFontSize` + `toastFontWeight` MUST control toast text typography.
- FR-05: `toastAnimationDuration` MUST control the slide-in and slide-out speed.
- FR-06: `toastBoxShadow` MUST override the hardcoded box-shadow.
- FR-07: `toastEnterFromBottom` MUST switch the toast's position from top-16px to bottom-16px and reverse the animation direction.
- FR-08: All 8 new settings MUST be persisted to the database and reflected in the public CSS delivery API.
- FR-09: The DCP preview for the Toasts sub-section MUST show the toast with all new properties applied in real time.
- FR-10: Existing `toastBgColor` and `toastTextColor` in `generalSettings` JSON blob MUST remain unchanged and backward-compatible.

## Non-Functional Requirements
- NFR-01: **Backward Compatibility** â€” existing `DesignSettings` rows (null for new columns) MUST fall back to the current hardcoded defaults via the CSS generator (e.g., null `toastBorderRadius` â†’ `8px`).
- NFR-02: **No migration** â€” all new columns are nullable with defaults; no `prisma migrate` data transform required.
- NFR-03: **Widget bundle rebuild required** â€” widget CSS files and JS must be rebuilt and deployed for changes to appear in the storefront.
- NFR-04: **Both widget types** â€” toast CSS exists identically in `bundle-widget.css` (product-page) and `bundle-widget-full-page.css`; both must be updated.

## User Stories
- As a merchant, I want to set the border-radius of my toast so I can make it pill-shaped or rectangular to match my brand.
- As a merchant, I want to add a border to my toast so it stands out against light backgrounds.
- As a merchant, I want to control the font size and weight in the toast so it matches my store typography.
- As a merchant, I want to set the animation duration so I can make the toast slide in slowly for a luxurious feel.
- As a merchant, I want the toast to slide up from the bottom so it doesn't overlap with a fixed header.
- As a developer/power user, I want to set a custom box-shadow on the toast for advanced depth effects.

## Scope
### In Scope
- 8 new DCP settings for toast appearance and animation
- Prisma schema additions (direct columns)
- TypeScript type additions
- CSS variable generator additions
- `mergeSettings` additions
- `handlers.server.ts` additions (in `extractGeneralSettings`)
- Widget CSS updates: `bundle-widget.css` + `bundle-widget-full-page.css`
- Widget JS update: `toast-manager.js` reads `--bundle-toast-enter-from-bottom`
- `ToastsSettings.tsx` UI updates
- `GeneralPreview.tsx` toast preview update
- Widget rebuild (version bump to 2.0.0)
- Issue tracking + commit

### Out of Scope
- Undo button typography customization
- Toast max-width customization
- Per-message toast customization
- Auto-dismiss duration DCP control (setTimeout remains hardcoded in JS)
- Toast position X-axis (horizontal center is always maintained)
