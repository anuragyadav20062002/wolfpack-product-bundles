# Business Requirement: Legacy Backwards-Compatibility Code Removal (Round 2)

## Executive Summary

Remove two remaining backwards-compatibility code paths that were previously preserved out of caution. The merchant has confirmed these can now be dropped, simplifying both the checkout extension and the widget messaging logic.

## Items in Scope

### 1. Checkout Component Parsing — Legacy Object Format
**File:** `extensions/bundle-checkout-ui/src/Checkout.tsx` (lines 56–76)

The `parseComponents` function currently supports two formats:
- **New (compact):** Compact arrays `[title, qty, retailCents, bundleCents, discountPct, savingsCents]` — written by the current cart transform
- **Legacy (object):** Named-property objects `{ title, quantity, retailPrice, ... }` — written by an older cart transform that is no longer deployed

The legacy branch is only reachable if `parsed[0]` is a plain object (not an array). All carts created by the current system always produce compact arrays.

### 2. Widget Messaging — `pricing.messages` Fallback Path
**Files:** `app/assets/bundle-widget-product-page.js` (lines 292–309), `app/assets/bundle-widget-full-page.js` (lines 433–449)

The `updateMessagesFromBundle()` method has three branches:
- **New:** `messaging.progressTemplate` / `messaging.successTemplate` — written by current metafield sync
- **Legacy:** `pricing.messages.progress` / `pricing.messages.qualified` — written by older metafield sync
- **Default:** show/hide flags based on `pricing.enabled`

The legacy branch is reachable only if a bundle's metafield was last written before the `messaging` top-level key was introduced. Dropping this path means such bundles fall through to the default behaviour (messaging shown if pricing is enabled, with widget default templates).

## Recommended Approach

Remove both legacy branches. The compact array format is the only format produced by the current cart transform. All new bundle saves produce the `messaging` top-level key. Any old bundle that falls through will get sensible default behaviour from the `else` branch.

## Functional Requirements

- FR-01: `parseComponents` in `Checkout.tsx` always treats parsed data as compact arrays; remove format detection and legacy object branch
- FR-02: `updateMessagesFromBundle` in `bundle-widget-product-page.js` removes `pricingMessages` variable and `else if` branch
- FR-03: `updateMessagesFromBundle` in `bundle-widget-full-page.js` removes `pricingMessages` variable and `else if` branch
- FR-04: Widget rebuilt after JS changes

## Non-Functional Requirements

- NFR-01: TypeScript compiles without new errors
- NFR-02: Widget behaviour unchanged for bundles with the new `messaging` metafield structure
- NFR-03: Old bundles without `messaging` fall through to the existing `else` default branch (no crash)
