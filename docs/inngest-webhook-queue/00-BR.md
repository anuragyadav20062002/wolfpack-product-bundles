# Business Requirement: Inngest Durable Webhook Queue

## Executive Summary

Replace the current fire-and-forget in-process webhook processing with Inngest — a managed durable event queue — so that any webhook event the processor fails to handle is automatically retried with exponential backoff and can be replayed on demand from a dashboard.

## Problem Statement

**Who is affected:** The engineering team and indirectly merchants whose data changes (inventory, subscriptions, product deletes) might silently not be reflected in the app.

**Current flow:**

```
Shopify POST /webhooks
  → HMAC validate
  → HTTP 200 returned to Shopify immediately
  → processPubSubMessage() runs asynchronously (fire-and-forget)
      → writes WebhookEvent row (processed: false)
      → runs handler (inventory sync, subscription update, etc.)
      → marks WebhookEvent.processed = true on success
```

**Problems with the current approach:**

1. **No automatic retry.** If the handler throws (DB timeout, Shopify API 429, cold-start OOM), the `WebhookEvent` row stays `processed: false` forever. There is no polling mechanism or retry scheduler to pick it up. Events are silently lost.

2. **No visibility.** There is no dashboard showing failed events, retry attempts, or processing history. The only way to detect failures is to grep logs or query the DB directly.

3. **No replay.** If a bug is deployed that corrupts handler output, there is no way to re-run all affected events after the fix is deployed. Each event would need to be manually re-triggered or re-ingested.

4. **In-process risk.** Processing runs inside the same Node.js process that received the HTTP request. If the worker process restarts (Render deploy, OOM), any in-flight jobs are lost silently.

5. **Growing unprocessed backlog is invisible.** The `WebhookEvent.processed = false` rows accumulate with no alert, no retry, and no expiry.

## Research Findings

### Approach A: BullMQ + Upstash Redis
- Self-hosted queue backed by Redis
- Requires Redis infrastructure (Upstash free: 10k commands/day, 256MB)
- Full control over retry policy, concurrency, rate limiting
- Needs a separate worker process to run the queue consumer
- No built-in replay dashboard without Bull Board (additional setup)
- **Pros:** Runs fully on-prem, cheap at scale, very flexible
- **Cons:** More infrastructure to manage; Upstash free tier is restrictive for high-volume; no managed replay UI

### Approach B: Inngest (managed)
- Fully managed event queue — no Redis, no worker process to run
- Free tier: 50,000 events/month, 3-day history, unlimited functions
- Built-in retry (up to 20 attempts, configurable backoff)
- Dashboard: event history, function run details, manual replay, live tail
- Integrates with Remix via a single `/api/inngest` serve route
- Events are sent via HTTP from existing webhook worker; Inngest calls back to the Remix app
- **Pros:** Zero infra overhead; generous free tier; replay UI out of the box; single npm package
- **Cons:** Inngest's callback must reach your Remix app (requires public URL — already true in production); functions run in your app's process (not a separate service)

### Approach C: Native PostgreSQL queue (pg-boss / Graphile Worker)
- Uses the existing Postgres DB as a queue
- No new infrastructure
- Mature libraries with retry support (pg-boss, Graphile Worker)
- **Pros:** No new services; reuses existing DB
- **Cons:** Polling-based (adds DB load); no managed dashboard; more boilerplate than Inngest

**Recommended: Approach B — Inngest**

Rationale:
- The app already has a Remix web service on Render with a public URL — Inngest callbacks work immediately
- Free tier covers the expected event volume comfortably (Shopify webhooks for a single app rarely exceed 10k/day)
- Built-in replay and live dashboard eliminates the entire "silent failure" visibility problem
- No new infrastructure to deploy or monitor
- Existing handler code (`subscription.server.ts`, `product.server.ts`, etc.) can be reused directly inside Inngest functions with zero changes

## Functional Requirements

- **FR-01:** Every Shopify webhook received by the webhook worker must be forwarded to Inngest as a named event before returning 200 to Shopify.
- **FR-02:** Each webhook topic must map to a dedicated Inngest function that executes the existing handler logic.
- **FR-03:** Failed Inngest function runs must be retried automatically (minimum 3 retries with exponential backoff).
- **FR-04:** Idempotency must be preserved — reprocessing the same webhook must not create duplicate side-effects (existing `WebhookEvent` dedup logic must be retained).
- **FR-05:** The Remix app must expose a `POST /api/inngest` route that serves as the Inngest function callback endpoint.
- **FR-06:** The Inngest signing key and event key must be stored in environment variables, not hardcoded.
- **FR-07:** Local development must work without a public URL, using Inngest Dev Server (`npx inngest-cli@latest dev`).

## Non-Functional Requirements

- **NFR-01: Latency** — Inngest event send (`inngest.send()`) must add < 50ms to webhook receipt time. The webhook worker must still return 200 to Shopify within the 5-second window.
- **NFR-02: Backward compatibility** — Existing `WebhookEvent` DB table and idempotency logic must continue to work. The Inngest layer sits on top, not replacing the DB record.
- **NFR-03: Observability** — All function executions (success, failure, retry) must be visible in the Inngest dashboard.
- **NFR-04: Graceful degradation** — If Inngest is unreachable when the webhook worker calls `inngest.send()`, the error must be logged and the webhook must still return 200 to Shopify (to prevent Shopify retries flooding the system).
- **NFR-05: Security** — The `/api/inngest` endpoint must validate the Inngest signature on every inbound request.

## User Stories (High Level)

- As an engineer, I want failed webhook events to be retried automatically so I don't have to manually re-trigger them after a transient error.
- As an engineer, I want to see the full execution history of every webhook event in a dashboard so I can debug failures without querying the DB.
- As an engineer, I want to replay a specific failed event after deploying a bug fix so affected merchants' data is corrected without requiring a re-send from Shopify.
- As an engineer, I want local development to work with a local Inngest Dev Server so I can test webhook processing without needing a public URL.

## Scope

### In Scope
- Integrate Inngest SDK into the Remix app and webhook worker
- Create one Inngest function per webhook topic group (subscription, product, inventory, orders, gdpr, lifecycle)
- Update webhook worker to send events to Inngest instead of (or in addition to) direct processing
- Add `/api/inngest` serve route to Remix app
- Environment variable configuration for INNGEST_EVENT_KEY and INNGEST_SIGNING_KEY
- Local dev setup instructions (Inngest Dev Server)

### Out of Scope
- Replacing or removing the existing `WebhookEvent` DB table or idempotency logic
- Building a custom admin UI for event management (Inngest dashboard is sufficient)
- Migrating historical unprocessed `WebhookEvent` rows to Inngest
- Custom retry policies per topic (all topics use the same default: 3 retries, exponential backoff)
- Inngest Crons / scheduled functions (separate feature)

## Risks

| Risk | Likelihood | Mitigation |
|------|------------|------------|
| Inngest unreachable when webhook received | Low | Wrap `inngest.send()` in try-catch; log error; still return 200 to Shopify; existing direct processing remains as fallback during transition |
| Free tier (50k events/month) exceeded | Low | Monitor in Inngest dashboard; at current scale well within limits |
| `/api/inngest` endpoint not publicly reachable on Render | Very Low | Render web services are always public; already confirmed by existing app traffic |
| Inngest callback hits Remix during deploy/restart | Low | Inngest retries failed callbacks automatically; no events lost |
| Duplicate processing if both direct path and Inngest path are active during transition | Medium | Existing `WebhookEvent` idempotency unique constraint prevents double side-effects |
