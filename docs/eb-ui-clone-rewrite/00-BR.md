# Business Requirement: EB Evidence UI Clone Rewrite

**Feature ID:** eb-ui-clone-rewrite
**Issue:** eb-ui-clone-rewrite-1
**Status:** Draft
**Created:** 2026-05-26
**Author:** Feature Pipeline

## Executive Summary

Wolfpack's FPB and PPB configure surfaces and storefront widgets must be rewritten to match the evidence-backed Easy Bundles audit rather than approximate earlier parity work. The rewrite is required because merchant-facing controls, saved payload contracts, template mappings, storefront behavior, and cart properties must line up with the audited competitor behavior before parity can be claimed.

## Business Context

Merchants compare the Wolfpack app against EB at the Admin configuration level and at the shopper storefront level. Partial resemblance is not enough: a control is only useful if merchants can find it in the same place, save it to the expected data contract, see the same storefront effect on desktop and mobile, and get the expected cart behavior.

The live audit in `docs/competitor-analysis/17-eb-complete-configure-e2e-audit.md` is the primary input. It includes Admin screenshots, network payloads, runtime JSON, desktop/mobile storefront screenshots, and cart proof for the clone-scoped surfaces. Emails and Customize Emails are excluded from this rewrite.

## Problem Statement

Current parity work is split across section-specific issues and does not provide one evidence gate that proves Admin UI, persistence, runtime, storefront, and cart behavior per control/template. This creates risk that a control looks correct but saves to the wrong field, or that a payload is correct but the storefront behavior remains incomplete.

## Goals

- Rebuild FPB and PPB configure/Admin UI as a shared clone-grade shell with evidence-backed section layouts and dependencies.
- Normalize direct persisted contracts for audited configuration fields without adding backwards-compatibility shims.
- Rewrite FPB and PPB storefront runtime behavior for the eight audited templates and their control-dependent states.
- Maintain a manifest that records evidence status per control/template before any parity claim.
- Keep screenshots and raw investigation artifacts out of the worktree.

## Non-Goals

- Emails and Customize Emails.
- Unsupported PPB Customize Colors & Language behavior beyond proven global Brand Config behavior.
- Guessing or filling evidence gaps from visual similarity alone.
- Adding new DCP controls before EB parity is proven. Existing DCP behavior should be preserved unless it blocks parity or causes breakage.
- Running `shopify app deploy` autonomously.

## Success Metrics

- Every evidence-backed manifest row is green for Admin visual parity, persistence, desktop storefront, mobile storefront, runtime JSON, and cart proof where relevant.
- Evidence-limited rows remain explicitly marked blocked with the missing proof named.
- Unit and integration tests cover serializers, mappings, dependency gates, and widget reducers before implementation.
- Widget source changes are built into Shopify extension assets and CSS size remains under Shopify limits.

## Constraints

- No competitor references in code identifiers, comments, filenames, or public app copy.
- No backwards-compatibility shims or legacy JSON fallbacks.
- Preserve the FPB widget load order: metafield cache first, proxy API fallback second, existing 503/504 retry retained.
- Custom non-Polaris Admin controls are allowed only under the narrow `eb-ui-clone-rewrite-1` exception in `AGENTS.md`.
- Screenshots remain under `/private/tmp/eb-complete-configure-audit-2026-05-25/` or another non-worktree evidence folder.
