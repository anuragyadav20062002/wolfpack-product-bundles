# Business Requirement: Theme Color Inheritance for Free Plan Bundle Widget

## Executive Summary

When a merchant is on the Free plan (DCP gated), their bundle widget should automatically inherit the store's active Shopify theme colors — mapping theme accent, button, text, and background colors to the bundle widget's global color anchors so the widget looks visually integrated with the store rather than using generic black/white defaults.

---

## Problem Statement

**Who experiences it:** Free plan merchants who have installed the bundle widget. Their bundles render with hardcoded defaults (`#000000` buttons, `#6B7280` secondary text, `#FFFFFF` backgrounds) that have no relationship to the store's design language. This creates a jarring disconnect between the bundle widget and the rest of the storefront.

**Pain points:**
1. Merchants on Free plan see generic-looking bundles that look like a third-party widget, not a native store element.
2. First impressions during trial/free period are worse than they should be — merchants may churn before even evaluating the Grow plan DCP.
3. Bundle conversion rates are likely lower when the widget looks visually mismatched.

**Expected outcome:** With zero configuration, a Free plan merchant's bundle widget buttons match their store's primary brand color, text matches their store's body text color, and backgrounds match their store's background palette.

---

## Research Findings

### How Bundle CSS Colors Currently Work

The bundle widget loads its styles from a dynamic CSS endpoint:
```
GET /apps/product-bundles/api/design-settings/{shopDomain}?bundleType=product_page|full_page
```

This endpoint generates ~150 CSS custom properties (`--bundle-*` vars) using `generateCSSFromSettings()`. Most color vars cascade from **6 global color anchors**:

| Anchor | Current Hardcoded Default | What It Controls |
|--------|--------------------------|------------------|
| `globalPrimaryButton` | `#000000` | Buttons, tabs, tier pills, step timeline, search focus, footer scrollbar, toasts |
| `globalButtonText` | `#FFFFFF` | Button labels, quantity selector text, header tab active text |
| `globalPrimaryText` | `#000000` | Product titles, prices, headings, header text, footer text |
| `globalSecondaryText` | `#6B7280` | Strike prices, empty state text, footer secondary |
| `globalFooterBg` | `#FFFFFF` | Footer background |
| `globalFooterText` | `#000000` | Footer text, back button text |

**Key insight:** If we correctly populate these 6 anchors with theme colors, the entire widget color scheme automatically inherits the store's design without touching any individual CSS variable.

### How Shopify Theme Colors Are Structured

Shopify themes expose their color palette in two ways:

1. **`config/settings_data.json`** (recommended): A JSON file in the active theme, containing the merchant's current color customizations. Readable via `GET /themes/{id}/assets.json?asset[key]=config/settings_data.json` (REST) or `OnlineStoreTheme.files()` query (GraphQL Admin API).

2. **Theme CSS variables** (e.g. `--color-accent-1`): Injected by the theme into the storefront DOM. Not reliably accessible to an app extension's shadow DOM.

**Approach A — Shopify Admin GraphQL `OnlineStoreTheme` + `files()` query**
- Fetch the active theme via `themes { roles { main { id } } }`
- Fetch `config/settings_data.json` via `theme.files(filenames: ["config/settings_data.json"])`
- Parse the JSON to extract color values
- **Pros:** Authoritative source, works for all theme types (Dawn, custom, headless)
- **Cons:** Requires `read_themes` Shopify API scope; 2 GraphQL calls per sync
- **Verdict:** ✅ Recommended

**Approach B — Liquid `settings.*` injection in the block**
- Read `{{ settings.colors_accent_1 }}` etc. directly in the Liquid block and inject a `<style>` tag that sets `--bundle-global-primary-button: {{ settings.colors_accent_1 }};`
- **Pros:** Zero API calls, no DB storage, always up-to-date
- **Cons:** Cannot conditionally apply only for Free plan (plan check is server-side). Cannot reflect in DCP preview. Brittle — only works for themes with standard `colors_*` keys (not all themes).
- **Verdict:** ❌ Rejected — cannot gate by plan, brittle for non-standard themes

**Approach C — Storefront CSS variable reading via widget JS**
- Widget JS reads computed CSS variables from the storefront DOM (`getComputedStyle(document.documentElement)`) and applies them to bundle vars
- **Pros:** Zero server changes
- **Cons:** Shadow DOM isolation means theme vars are not accessible inside the widget. Unreliable.
- **Verdict:** ❌ Rejected — shadow DOM breaks CSS inheritance

### Color Mapping (Shopify `settings_data.json` → Bundle Anchors)

Dawn and most modern Shopify themes use these standard color keys in `settings_data.json > current > colors`:

| Shopify Key | Meaning | Maps To |
|------------|---------|---------|
| `colors_accent_1` | Primary brand/action color | `globalPrimaryButton` |
| `colors_solid_button_labels` | Text on filled buttons | `globalButtonText` |
| `colors_text` | Primary body text | `globalPrimaryText` |
| `colors_secondary_button_labels` | Secondary / muted text | `globalSecondaryText` |
| `colors_background_1` | Page background color | `globalFooterBg` |
| `colors_text` | Same as primary text for footer | `globalFooterText` |

**Fallback:** If a theme doesn't define these keys (older/custom themes), fall back to the existing hardcoded defaults (`#000000`, `#FFFFFF`, etc.).

### Where to Store Theme Colors

Theme colors must be cached server-side because:
1. The CSS endpoint is loaded unauthenticated by a `<link>` tag in Liquid — it cannot make Shopify API calls on the fly.
2. We need the colors available in the DCP preview route too.

**Recommended storage:** New `themeColors` JSON field on the `DesignSettings` model (one entry per shop per bundle type). Populated on:
- App install (session token flow gives us Admin API access)
- "Sync Bundle" action (already an existing UX touchpoint in the app)

**Staleness:** Theme colors change infrequently (merchant re-themes their store). Sync-on-demand is sufficient. No polling or webhooks needed for MVP.

## Functional Requirements

- **FR-01:** When a merchant is on the Free plan, the bundle widget's CSS uses the stored Shopify theme colors as the 6 global color anchor values instead of hardcoded defaults.
- **FR-02:** When a merchant is on the Grow plan, the 6 global anchors continue to use the DCP-configured values (or hardcoded defaults if no DCP customization has been saved).
- **FR-03:** Theme colors are fetched from the active Shopify theme via Admin GraphQL API and stored in a `themeColors` JSON field on `DesignSettings`.
- **FR-04:** Theme colors are synced on app install and when the merchant triggers "Sync Bundle" from any bundle configure page.
- **FR-05:** If no theme colors are stored (pre-existing install before this feature), the CSS falls back to the existing hardcoded defaults.
- **FR-06:** If the active theme does not define the expected color keys, fall back to existing hardcoded defaults for each missing color individually.
- **FR-07:** The DCP preview pane for Free plan merchants reflects the theme-inherited colors (the preview uses the same CSS endpoint, so this is automatic).
- **FR-08:** The `read_themes` Shopify scope is added to the app's scope list if not already present.

---

## Non-Functional Requirements

- **NFR-01:** The CSS endpoint response time must not regress — theme colors are read from DB, not fetched from Shopify API on each request.
- **NFR-02:** Backward compatible — existing installs without `themeColors` stored gracefully fall back to current defaults. No migration needed.
- **NFR-03:** No new DB migrations required beyond adding a nullable JSON column to `DesignSettings`.
- **NFR-04:** The theme color sync function is idempotent — calling it multiple times produces the same result.
- **NFR-05:** Scope change (`read_themes`) must be handled per Shopify's scope re-request flow — existing merchants prompted to re-authorize.

---

## User Stories (High Level)

- **As a Free plan merchant**, I want my bundle widget to automatically match my store's brand colors so that my bundles look native to my storefront without any setup effort.
- **As a Free plan merchant**, I want the bundle colors to update when I re-sync my bundle after changing my store theme, so my bundles stay in sync with my branding.
- **As a Grow plan merchant**, I want my custom DCP color choices to be unaffected by any theme color inheritance logic, so my carefully tuned design settings are preserved.
- **As the app owner**, I want theme color inheritance to be a compelling visual reason for Free plan merchants to keep using the app, increasing conversion to Grow plan.

---

## Scope

### In Scope
- Fetching active theme colors via Shopify Admin GraphQL (REST fallback acceptable)
- Storing colors in a new `themeColors` JSON field on `DesignSettings`
- Applying theme colors as global anchor fallbacks for Free plan merchants in the CSS endpoint
- Triggering color sync on install + "Sync Bundle"
- Adding `read_themes` scope

### Out of Scope
- Real-time theme change detection (no webhooks for MVP)
- Per-color-group theme inheritance (e.g. "only inherit button colors") — apply all 6 anchors or none
- Grow plan merchants seeing theme colors as defaults in the DCP picker — Grow plan continues with existing behavior
- Font family inheritance from theme

---

## Risks

| Risk | Likelihood | Mitigation |
|------|-----------|------------|
| Theme uses non-standard color keys | Medium | Per-key fallback to hardcoded defaults |
| `read_themes` scope triggers re-auth friction for existing merchants | Medium | Document in release notes; scope request is a standard Shopify flow |
| Theme colors are invalid CSS values (malformed hex) | Low | Validate each color value before storing; discard invalid ones |
| DesignSettings record doesn't exist for a shop (never visited DCP) | Medium | Upsert on sync; CSS endpoint already handles missing DesignSettings |
