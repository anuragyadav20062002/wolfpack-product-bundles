# Business Requirement: FPB Bundle Config Metafield Cache

## Executive Summary
Full-page bundle (FPB) widgets currently depend entirely on the Shopify app proxy to fetch bundle
configuration at runtime. If the proxy is not registered or temporarily unavailable, every
storefront visitor on every FPB page sees a broken widget — with no fallback and no merchant alert.

## Problem Statement
**Who:** Merchants using full-page bundles on their storefront. Storefront visitors browsing those pages.

**Pain point:** The FPB widget calls `/apps/product-bundles/api/bundle/{id}.json` via the Shopify
app proxy on every page load. This proxy must be registered by deploying the app. If a store is
on a fresh install, the proxy registration has broken, or Shopify's propagation has not completed,
the proxy returns 404 — and the widget displays "Bundle unavailable" to every customer. Merchants
have no visibility into this until a customer complains.

**Contrast with PDP bundles:** Product-page bundles store the full bundle config in a Shopify
product metafield (`$app:bundle_ui_config` on the variant). The widget reads it from
`data-bundle-config` injected by Liquid — zero proxy calls required. PDP bundles are therefore
completely resilient to proxy failures.

## Research Findings

- **Approach A: Metafield cache on the Shopify page** — Write the serialised bundle config as
  a JSON metafield on the Shopify page object when "Add to storefront" or "Sync" runs. The FPB
  Liquid template reads it and injects it as `data-bundle-config`. The widget prefers this over the
  API call. Falls back to the proxy API if the metafield is absent (handles old pages created before
  this feature).
  - Pros: Mirrors the proven PDP pattern exactly; zero proxy dependency on hot path; works
    immediately after page creation; survives proxy outages; no extra latency for storefront visitors.
  - Cons: Metafield can become stale if the bundle changes and the merchant doesn't re-sync. However,
    the existing "Sync Bundle" button already resets stale state — the same UX applies here.
  - **Recommended.**

- **Approach B: Embed config in an inline `<script>` tag** — Have the Liquid template call the
  Shopify Admin API at render time (not possible from Liquid) or use a server-side tag. Not feasible
  in a theme extension block — Liquid cannot make server-side API calls.
  - Rejected: Not possible within Shopify's Liquid execution model.

- **Approach C: Service worker or localStorage cache** — Cache the API response client-side after
  the first successful proxy call.
  - Rejected: Still requires at least one successful proxy call; doesn't fix the root cause; adds
    significant complexity; cache invalidation is hard.

**Recommended:** Approach A — metafield cache, matching the PDP pattern.

## Functional Requirements

- **FR-01:** When a merchant clicks "Add to storefront" on the FPB configure page, the full bundle
  config is serialised to JSON and written to a page metafield (`custom:bundle_config`, JSON type,
  PUBLIC_READ storefront access).
- **FR-02:** When a merchant clicks "Sync Bundle" on the FPB configure page, the same metafield is
  updated with the latest bundle config.
- **FR-03:** The FPB Liquid template reads `page.metafields.custom.bundle_config` and injects it
  as `data-bundle-config` on the widget container div.
- **FR-04:** The FPB widget checks for `data-bundle-config` first. If present and valid JSON, it
  uses it directly (no proxy API call). If absent, it falls back to the existing proxy API call.
- **FR-05:** The metafield definition (`custom:bundle_config`, JSON, PUBLIC_READ) must exist on the
  store for Liquid to read it reliably. The app writes this definition on first use.

## Non-Functional Requirements

- **NFR-01 Performance:** Storefront page load eliminates the proxy API fetch on the critical path.
  No extra latency for storefront visitors.
- **NFR-02 Backward Compatibility:** Existing FPB pages that have `bundle_id` set but no
  `bundle_config` metafield must continue to work via the proxy API fallback — zero merchant action
  required until they next sync.
- **NFR-03 Data freshness:** The metafield reflects the bundle config at the time of last
  "Add to storefront" or "Sync". Merchants are informed via existing sync UX that re-syncing
  propagates changes to the storefront.
- **NFR-04 No DB migration:** No Prisma schema changes. Bundle config is derived at write-time from
  existing DB fields.

## User Stories (high level)

- As a merchant, I want my full-page bundle pages to load correctly for customers even if the app
  proxy is temporarily unavailable.
- As a merchant, when I sync my bundle, I want the storefront page to immediately reflect the
  latest product steps and pricing.
- As a developer, I want FPB to follow the same resilient data pattern as PDP bundles.

## Scope

### In Scope
- Write `custom:bundle_config` metafield to Shopify page in `handleWidgetPlacement` and
  `handleSyncBundle`.
- Read and inject the metafield in `bundle-full-page.liquid`.
- Update FPB widget to prefer `data-bundle-config` over proxy API call.
- Register the metafield definition (namespace + key + type + access) alongside the existing
  `bundle_id` definition.

### Out of Scope
- Changing the proxy API endpoint (it remains as the fallback and for future tier-switching).
- Migrating old pages automatically (handled by fallback path + merchant re-sync).
- PDP bundle changes (already working correctly).
- Widget CSS changes.

## Risks

| Risk | Likelihood | Mitigation |
|------|------------|------------|
| Metafield value size limit (Shopify JSON metafield limit is 128 KB) | Low — typical bundle config is 5–30 KB | Truncate step products list gracefully; log a warning if approaching limit |
| Stale metafield after bundle changes | Medium | "Sync Bundle" button updates it; show info notice on configure page that changes require a sync to go live |
| Metafield definition creation race condition | Low | Use upsert pattern (define-then-write); idempotent on repeat |
