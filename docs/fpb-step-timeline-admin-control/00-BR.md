# Business Requirement: Admin-Controlled Step Timeline Visibility for Full-Page Bundles

## Executive Summary

Merchants using pricing tiers (flat-grid BYOB) currently must navigate to the Shopify Theme
Editor to hide the step timeline. Add a "Show step timeline" toggle directly in the Admin UI
pricing tiers section, stored in the database, so the widget respects it without theme editor
intervention. Guard against the confusing steps + tiers combination via a warning modal.

## Problem Statement

When a merchant configures a flat-grid bundle (e.g., "Buy 2 @ ₹499, Buy 3 @ ₹699") using
pricing tier pills, the step progression timeline is irrelevant and visually cluttering. To hide
it they must: (1) open Theme Editor, (2) find the bundle-full-page block, (3) uncheck
"Show step timeline". That is 3+ clicks outside the app. The Admin UI should own this setting.

Additionally, if a merchant accidentally configures both multi-step bundle logic AND pricing
tier pills together, the resulting UX is confusing (steps imply sequential navigation; tiers
imply a flat single-screen grid). A warning should surface this conflict proactively.

## Research Findings

### Current architecture
- `showStepTimeline` lives only as a theme editor block setting (`data-show-step-timeline`)
- `tierConfig` is stored in `Bundle.tierConfig` (JSON field) and returned via API
- Widget resolves tierConfig: API (preferred) → data attribute (fallback)
- Widget reads showStepTimeline: data attribute only (no API source)
- Steps + pricing tiers are configured on the same route page (different tabs)

### Approach A: Nullable DB field + API propagation (Recommended ✅)
Store `showStepTimeline Boolean?` on the Bundle model.
- `null` = not set by admin → widget falls back to data attribute (full backward compat)
- `true` / `false` = admin override → widget uses this, ignores data attribute
Pros: backward compatible, no migration impact on existing bundles, no theme editor conflicts
Cons: one nullable field, slightly more complex widget resolution logic

### Approach B: Non-nullable with default true
Store `showStepTimeline Boolean @default(true)`
Pros: simpler schema
Cons: BREAKS existing merchants who set `show_step_timeline = false` in Theme Editor,
because on migration day all bundles get `true` which the widget would now prefer over data attr.

## Functional Requirements

- FR-01: A "Show step timeline" checkbox appears in the Pricing Tiers section when the
  merchant has configured ≥ 2 tiers (i.e., tier pills are active on the storefront).
- FR-02: The setting is saved to `Bundle.showStepTimeline` (nullable boolean) in the DB.
- FR-03: The API endpoint returns `showStepTimeline` (null | true | false).
- FR-04: The widget resolves showStepTimeline: API non-null value → data attribute fallback.
- FR-05: On tier switch, the new bundle's `showStepTimeline` is applied immediately on re-render.
- FR-06: When the merchant removes tiers to < 2, the server resets `showStepTimeline = null`
  so the theme editor data attribute regains control.
- FR-07: A warning modal appears when: (a) tiers reach ≥ 2 and the bundle has > 1 step,
  or (b) a step is added while ≥ 2 tiers are configured. Warning is advisory — merchant
  can dismiss and proceed.

## Non-Functional Requirements

- NFR-01: Zero regressions for existing merchants using only theme editor toggle.
- NFR-02: Zero regressions for merchants with < 2 tiers (checkbox never shown, DB null).
- NFR-03: Widget must correctly apply showStepTimeline after every tier switch.
- NFR-04: DB migration uses nullable field (no default) — no data backfill required.

## Scope

### In Scope
- `Bundle` schema: add `showStepTimeline Boolean?`
- `api.bundle.$bundleId[.]json.tsx`: return the new field
- Widget: resolve from API + apply after tier switch
- Configure route: state + formData + checkbox UI
- Handler: parse + reset to null when < 2 tiers
- Warning modal: tiers tab + steps tab trigger
- Tests: all new logic paths

### Out of Scope
- Product-page bundles (showStepTimeline is full-page only)
- Theme editor block TOML changes (existing checkbox stays)
