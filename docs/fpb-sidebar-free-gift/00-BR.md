# Business Requirement: FPB Sidebar Panel Redesign + Free Gift + Default Product

## Executive Summary
Revamp the full-page bundle widget's right sidebar panel to match the design quality and feature set of Skai Lama (Easy Bundle Builder), including free gift step support and mandatory default product support — two features merchants expect and our primary competitor already ships.

## Problem Statement
Our full-page bundle widget sidebar exists but lacks two high-value features merchants need:

1. **Free Gift Step**: Merchants want to offer a free product (e.g., "Get a free cap when you buy 3 items") as a bundle incentive. Skai Lama ships this as a first-class feature. We have no support for it at all — no DB fields, no API fields, no widget rendering.

2. **Mandatory Default Product**: Merchants want to pre-include a product in every bundle (e.g., a "Gift Box" packaging product) that the customer cannot remove. Skai Lama's Bundle 39 demo ships this — "Gift Box $4.99" is always in the bundle from the start. We have `isDefault` and `defaultVariantId` defined in `BundleUiStep` type but no DB persistence, no API return, and no widget rendering.

3. **Sidebar Visual States**: Our sidebar lacks visual states Skai Lama's implementation has — locked/unlocked free gift section, skeleton empty slots for unfilled positions, "Free" badge on product cards in the free gift step.

4. **Mobile Bottom Bar**: Our mobile UX for the sidebar is unclear. Skai Lama's mobile converts the sidebar to a sticky bottom bar with `^N items` toggle that expands a full bundle panel.

Without these features, merchants evaluating us against Skai Lama will choose Skai Lama.

## Research Findings

### Competitive Analysis (Skai Lama)
Full analysis at `docs/skai-lama-analysis/fpb-sidebar-panel-analysis.md`.

Key patterns observed:
- **Free gift step**: Last step in the bundle, locked until N paid steps complete, products show $0.00 with original price strikethrough and a "Free" badge on the card
- **Sidebar locked state**: 🔒 icon + "Add N more product(s) to claim a FREE [giftName]!" in a gray bordered box
- **Sidebar unlocked state**: Green box "Congrats you are eligible for a FREE [item]!" with ✅
- **Sidebar complete + free gift selected**: Item shows in selected list at $0.00, original price struck through
- **Default product**: Gift Box appears in sidebar immediately on page load, cannot be removed, counts toward total
- **Mobile**: Sidebar replaced by sticky bottom bar; `^N items` caret + total + Next pill CTA; tap caret → bottom sheet slides up with full bundle details

### Current State
- `BundleStep` Prisma model: no `isFreeGift`, `isDefault`, `freeGiftName` fields
- `BundleUiStep` type (`app/services/bundles/metafield-sync/types.ts`): has `isDefault`, `defaultVariantId`, `discountBadgeLabel`, `categoryImageUrl` but NOT `isFreeGift` or `freeGiftName`
- API (`api.bundle.$bundleId.json.tsx`): returns only DB fields; new fields need to be added after DB migration
- Widget (`app/assets/bundle-widget-full-page.js`): no `isDefault`, `isFreeGift`, or free gift rendering anywhere

### Approach Options

**Option A: DB-first (Add Prisma fields → API → Widget)**
- Add `isFreeGift`, `isDefault`, `defaultVariantId`, `freeGiftName` to `BundleStep` schema
- Migrate DB, update API response, add admin UI fields, update widget
- Pros: Clean data model, persistent, queryable
- Cons: Requires DB migration, admin UI changes, more code surface

**Option B: Computed/convention-based (Widget infers from existing fields)**
- Free gift step = last step where `minQuantity: 0` (or step has discount with 100% off)
- Default product = step with `minQuantity: 0` and a single pre-configured StepProduct
- No DB changes needed
- Pros: Zero migration risk, faster to ship
- Cons: Fragile convention, not explicit, hard to extend, confusing for merchants

**Option C: Hybrid — DB fields for explicit flags, widget infers from pricing for free gift display**
- Add `isFreeGift Boolean` and `isDefault Boolean` + `defaultVariantId String?` to DB
- Free gift UI (locked/unlocked state) computed in widget from whether all non-free-gift steps are complete
- Default product auto-selected in widget without merchant interaction
- Recommended ✅

**Recommended: Option C (Hybrid)**
- Explicit flags (`isFreeGift`, `isDefault`) in DB give merchants clear admin UX
- Widget computes unlock state from `selectedCount >= requiredCount` (all non-free-gift steps satisfied)
- `defaultVariantId` lets merchants pin a specific variant as pre-selected
- Clean separation: data model is explicit, presentation logic in widget

## Functional Requirements

- FR-01: Merchant can mark a bundle step as `isFreeGift: true` via admin UI
- FR-02: Merchant can set a `freeGiftName` display label for the free gift section (e.g., "cap", "greeting card")
- FR-03: Merchant can mark a step as `isDefault: true` and set a `defaultVariantId` to pre-select a specific variant
- FR-04: Widget sidebar shows locked free gift section while required paid steps are not complete
- FR-05: Widget sidebar shows unlocked (green) free gift section once all non-free-gift steps are satisfied
- FR-06: Free gift step product cards show a "Free" badge label and $0.00 price
- FR-07: Free gift step page shows a custom heading (e.g., "Complete the look and get a cap free!")
- FR-08: Free gift item in sidebar selected list shows $0.00 with original price struck through
- FR-09: If user removes a paid item, free gift section re-locks
- FR-10: Default product step is pre-selected on widget load, shown in sidebar with no remove button
- FR-11: Default step is not shown as a navigable step in the step timeline (or shown as greyed-out / auto-completed)
- FR-12: Mobile: sidebar converts to sticky bottom bar with item count, total, and Next/Add To Cart button
- FR-13: Mobile: tapping the bottom bar expands a full bundle panel (bottom sheet) showing all selected items + free gift state
- FR-14: Skeleton empty slot placeholders shown in sidebar for unfilled step positions

## Non-Functional Requirements
- NFR-01: Zero breaking change to existing bundles — `isFreeGift` and `isDefault` default to `false`
- NFR-02: Widget must not break on bundles without free gift or default steps
- NFR-03: CSS size must remain under 100,000 B (Shopify app block limit) — check after any CSS changes
- NFR-04: Widget bundle build (`npm run build:widgets`) must succeed without warnings/errors

## User Stories (high level)
- As a merchant, I want to add a free gift step to my bundle so I can incentivize customers to buy more
- As a merchant, I want to pre-include a mandatory product (e.g., gift box) in every bundle so customers always receive it
- As a shopper, I want to see clearly that I've unlocked a free gift so I'm excited to complete my selection
- As a shopper on mobile, I want to see my bundle summary without needing to scroll up so I always know what I've selected

## Scope
### In Scope
- DB: add `isFreeGift`, `isDefault`, `defaultVariantId`, `freeGiftName` to `BundleStep`
- API: return new fields in bundle JSON endpoint
- Widget JS: free gift locked/unlocked states, free gift badge on cards, default product auto-selection, mobile bottom bar, skeleton slots
- Widget CSS: all new component styles (must stay under 100KB)
- Admin UI: toggle fields for `isFreeGift` and `isDefault` per step

### Out of Scope
- Multiple free gift steps per bundle (V1: max 1 free gift step)
- Free gift quantity > 1 (V1: always exactly 1 free item)
- Free gift without completing ALL paid steps (unlock condition is always "all other steps done")
- Per-step banner image (separate feature)
- Spring-physics toast animation (separate)
- Skai Lama bottom-sheet PDP widget redesign (separate, already tracked in existing issue)

## Risks
| Risk | Likelihood | Mitigation |
|------|------------|------------|
| DB migration fails on existing bundle data | Low | `isFreeGift` and `isDefault` default false — safe |
| CSS size exceeds 100KB after new styles | Medium | Run wc -c check before deploy (per CLAUDE.md) |
| Widget breaks on bundles without free gift | Low | All new code guarded by `step.isFreeGift` checks |
| Admin UI adds complexity merchants don't understand | Medium | Clear labels + help text in Polaris UI |
