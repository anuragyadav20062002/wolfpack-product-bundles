# Business Requirement: Image Upload in Promo Banner File Picker

**Feature ID:** promo-banner-upload
**Created:** 2026-02-20
**Status:** BR Complete → Handoff to PO

---

## Executive Summary

Merchants using the promo banner background image picker must currently upload images
separately via Shopify Admin → Content → Files before they can select them in the DCP.
This feature adds an "Upload image" action directly inside the FilePicker modal so the
entire workflow — upload and select — happens without leaving the DCP.

---

## Problem Statement

The existing `FilePicker` modal (shipped in `promo-banner-bg-image`) lets merchants browse
and pick images already in their Shopify Files library. However, if a merchant has a new
image they haven't uploaded yet, they must:

1. Leave the DCP
2. Navigate to Shopify Admin → Content → Files
3. Upload the image there
4. Return to the DCP and re-open the picker

This context switch is friction that slows the design configuration flow. The merchant's
expectation is that they can do everything in one place.

**Who experiences this:** Merchants setting up the promo banner for the first time, and
merchants who want to experiment with different images quickly.

---

## Research Findings

All technical research was completed and confirmed feasible prior to this pipeline.

### Approach A — Client-side direct upload (browser → Shopify staged URL)

Merchant picks a file → browser `fetch` POSTs directly to Shopify's staged upload URL.

**Pros:** No server round-trip for the binary; simpler streaming.
**Cons:** The Shopify `/admin/tmp/files` staged upload endpoint has **no CORS headers**.
All browsers will block the request when made from the embedded app iframe.
Additionally, the Admin GraphQL mutations (`stagedUploadsCreate`, `fileCreate`) require
the `X-Shopify-Access-Token` header, which must never be sent from the browser.

**Verdict:** ❌ Rejected — CORS-blocked; access token cannot be exposed to the browser.

---

### Approach B — Fully server-side upload via Remix action (Recommended ✅)

Merchant picks a file → browser multipart-POSTs to a Remix action route → Remix server
performs all three Shopify steps server-side (stagedUploadsCreate → binary upload →
fileCreate) → polls for READY status → returns the final CDN URL to the browser.

**Pros:**
- Fully CORS-safe (binary never leaves the server to a cross-origin endpoint).
- Admin access token stays server-side.
- Follows the exact pattern already used by `app/routes/app/app.bundles.cart-transform.tsx`
  and other routes that call `admin.graphql()`.
- Remix `action` handler keeps the code co-located and testable.

**Cons:**
- Binary passes through our server (latency proportional to file size; negligible for
  typical banner images ≤ 5 MB).
- Polling adds 2–10 seconds of wait time after upload. A progress indicator is required.

**Verdict:** ✅ Recommended.

---

### Approach C — App Proxy relay (client → app proxy → Shopify staged URL)

Use Shopify's app proxy to relay the upload from the browser through the proxy server.

**Pros:** Conceptually direct.
**Cons:** App proxy is designed for storefront-facing Liquid integration, not for Admin
API calls. The proxy cannot inject the admin access token. This approach is architecturally
wrong for admin operations.

**Verdict:** ❌ Rejected — wrong tool for admin API access.

## Functional Requirements

- **FR-01:** The FilePicker modal must include an "Upload image" button that opens a
  native file input accepting `image/*` MIME types.
- **FR-02:** When a file is selected, it is uploaded to our Remix server action via
  multipart form and forwarded to Shopify Files using the staged upload API.
- **FR-03:** The upload action must call `stagedUploadsCreate`, POST the binary to the
  staged URL, then call `fileCreate` to register the file in Shopify Content → Files.
- **FR-04:** The action must poll `node(id:)` until `fileStatus === "READY"` (up to 30s)
  before returning the CDN URL.
- **FR-05:** During upload and processing, the modal must show a progress/loading state
  with a descriptive label ("Uploading…" then "Processing…").
- **FR-06:** On success, the newly uploaded image is added to the file grid and
  pre-selected (highlighted with the blue border ring), ready for the merchant to confirm
  with "Select".
- **FR-07:** On failure (CORS error impossible server-side; FAILED status; timeout), a
  clear error message is shown inside the modal. The upload state resets so the merchant
  can retry.
- **FR-08:** The `write_files` access scope must be added to `shopify.app.toml`.
  Merchants already granted access will receive the new scope on next auth flow or at
  next token refresh — no action required from existing merchants.
- **FR-09:** File size must be validated client-side (≤ 20 MB) before sending to the
  server, to provide immediate feedback.
- **FR-10:** Only image MIME types are accepted (`image/jpeg`, `image/png`, `image/webp`,
  `image/gif`, `image/svg+xml`, `image/avif`).

---

## Non-Functional Requirements

- **NFR-01 Security:** Access token is never sent to the browser. Binary passes through
  the Remix server. No cross-origin token exposure.
- **NFR-02 Performance:** Typical banner images are 200 KB – 2 MB. Server relay adds
  negligible latency. Shopify processing typically completes within 2–10 seconds.
- **NFR-03 Scope change:** Adding `write_files` requires a scope update commit + deploy.
  This is a non-breaking scope addition (not removal).
- **NFR-04 Backward compatibility:** The existing FilePicker browse/select flow is
  unchanged. Upload is additive.
- **NFR-05 Timeout:** If Shopify does not return `READY` within 30 seconds (15 × 2s
  polls), surface a "Processing is taking longer than expected" message. The file will
  eventually appear in the Files library even if we timeout.

---

## User Stories (high level)

- As a **merchant**, I want to upload a new image directly from the promo banner picker
  so that I don't have to navigate away from the DCP.
- As a **merchant**, I want to see upload progress so that I know the image is being
  processed and I don't have to guess when to click "Select".
- As a **merchant**, I want clear error feedback if the upload fails so that I can retry
  without confusion.

---

## Scope

### In Scope
- "Upload image" button inside the existing `FilePicker` modal.
- New Remix action route: `app/routes/app/app.upload-store-file.tsx`.
- `write_files` scope addition to `shopify.app.toml`.
- Client-side file type and size validation.
- Progress/loading UX states ("Uploading…", "Processing…").
- Polling for READY status (max 30s); timeout message on overflow.
- Auto-select and display of the uploaded file in the picker grid on success.

### Out of Scope
- Multiple simultaneous file uploads.
- Drag-and-drop upload (future enhancement).
- Upload progress bar with byte-level percentage (streaming not available via standard
  Remix form action; showing spinner + label is sufficient).
- File management (rename, delete) inside the picker.
- Upload from URL or external source.

---

## Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| Shopify `READY` polling times out (>30s) | Low | Show timeout message; file still lands in Files library |
| `write_files` scope triggers re-auth for some merchants | Low | Non-breaking scope addition; existing tokens typically remain valid |
| Large files (5–20 MB) cause Remix action timeout | Low | Remix default timeout is 30s+; typical images are <5 MB |
| Shopify staged URL returns non-200 | Low | Server checks HTTP status; returns error JSON to client |
