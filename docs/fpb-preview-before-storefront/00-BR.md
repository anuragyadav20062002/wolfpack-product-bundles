# Business Requirement: Full-Page Bundle Pre-Storefront Preview

## Executive Summary

Merchants building full-page bundles currently cannot see how their bundle looks on the storefront until after they commit to adding it live. This creates unnecessary friction and reduces merchant confidence — a "Preview" button available before publishing resolves this.

---

## Problem Statement

**Who experiences it:** Merchants who are building or editing a full-page bundle and want to validate the visual result before exposing the page to shoppers.

**Current pain:**
- The first time a merchant can see their bundle rendered on the storefront is after clicking "Add to Storefront", which creates a live, publicly accessible Shopify page immediately.
- If the bundle needs adjustments (layout, product selection, copy), the merchant must make changes to a page that is already visible in the store's navigation or shareable by URL.
- There is no safe, low-commitment way to preview the bundle in the real storefront theme before publishing.
- Product-page bundles already have preview support via Shopify's `onlineStorePreviewUrl`. Full-page bundles have no equivalent.

**Business impact:** Merchants who are unsatisfied with the bundle's appearance after publishing are more likely to delete the page, creating orphaned bundle records. The absence of preview is a confidence barrier that slows the time-to-publish for new bundles.

---

## Research Findings

### Approach A: In-App Embedded Preview (iframe)
- Render the widget HTML at an app proxy route and embed it in a Polaris `Modal` or full-screen overlay.
- **Pros:** No Shopify page or asset required; instant.
- **Cons:** Renders in isolation without the merchant's real theme (fonts, colours, navigation). Not a representative preview. Significant custom implementation work.
- **Verdict:** ❌ Rejected — the preview would be misleading because it lacks theme context.

### Approach B: Theme Editor Deep-Link with previewPath
- Construct a Shopify Admin theme editor URL pointing to `pages/{handle}` using the `previewPath` parameter.
- **Pros:** Opens the real theme editor with theme styles applied.
- **Cons:** Requires the page to already exist in Shopify; the theme editor URL is not always stable across API versions; the merchant sees the theme editor chrome rather than a clean storefront preview.
- **Verdict:** ❌ Rejected — still requires page creation first and is a worse UX than a direct storefront preview.

### Approach C: Persistent Shared Preview Page
- Create one single draft Shopify page per shop (handle: `bundle-preview-app`). Before opening preview, update its metafields to point to the current bundle, then return `shareablePreviewUrl`.
- **Pros:** Only ever one extra Shopify page. No DB schema change.
- **Cons:** Metafield update before every preview open adds latency. Two merchants (same shop, two browser tabs) would conflict if they previewed different bundles simultaneously. Feels fragile.
- **Verdict:** ❌ Rejected — concurrency risk and per-preview latency are not worth the schema simplicity.

### Approach D: Draft Shopify Page Promoted to Published on "Add to Storefront" ✅ Recommended
- When "Preview" is clicked (and no storefront page yet exists), create a **draft (unpublished)** Shopify page with the bundle's template and metafields.
- Query `shareablePreviewUrl` from the Admin API — this URL is accessible to anyone with the link, regardless of the page's published state.
- Store the draft page GID in a new `shopifyPreviewPageId` field on the Bundle model.
- Subsequent "Preview" clicks re-query `shareablePreviewUrl` for the existing draft page (idempotent, no new page created).
- When "Add to Storefront" is clicked, **promote the draft page to published** (via `pageUpdate` with `isPublished: true`) rather than creating a new page.
- **Pros:** One page per bundle total (draft becomes published). Uses Shopify's native preview infrastructure. True storefront rendering with real theme. Clean DB tracking. Congruent with how product-page bundles use `onlineStorePreviewUrl`. No page duplication.
- **Cons:** Requires a small Prisma schema change (`shopifyPreviewPageId`, `shopifyPreviewPageHandle`). The `createFullPageBundle` service must be split into draft-create and publish steps.
- **Verdict:** ✅ Recommended.

## Functional Requirements

- **FR-01:** A "Preview" button must appear on the full-page bundle configure page when `shopifyPageHandle` is null (bundle not yet on storefront).
- **FR-02:** The first click of "Preview" (no draft exists) creates a draft Shopify page with the bundle template and metafields, queries `shareablePreviewUrl`, and opens it in a new tab.
- **FR-03:** Subsequent "Preview" clicks (draft exists, not yet published) re-query `shareablePreviewUrl` and open it without creating a new page.
- **FR-04:** "Add to Storefront" promotes the existing draft page to published if a draft exists, or falls back to creating a new published page if no draft exists.
- **FR-05:** After "Add to Storefront" completes, the UI transitions to the existing "View on Storefront" button (current behaviour, unchanged).
- **FR-06:** Bundles that already have `shopifyPageHandle` set (already on storefront) are unaffected — they show only "View on Storefront" with no change.
- **FR-07:** The draft page must have the same `templateSuffix`, bundle metafields, and bundle config metafield as a published page so the preview is accurate.
- **FR-08:** If preview page creation fails, a clear error toast is shown and no DB state is written.

## Non-Functional Requirements

- **NFR-01 Performance:** The first-click preview page creation should complete within the existing `validateWidgetPlacement` response time envelope (~3–5 s). Re-open previews must be near-instant (single GraphQL query).
- **NFR-02 Backward compatibility:** All existing bundles with `shopifyPageHandle` set must continue to work without modification. No migration required.
- **NFR-03 No backwards-compat shims:** Per CLAUDE.md rules, no fallback shims for old data. New fields default to null; existing bundles are unaffected.
- **NFR-04 Idempotency:** Clicking "Preview" multiple times must never create more than one draft preview page per bundle.
- **NFR-05 Cleanup:** If a merchant deletes the bundle before clicking "Add to Storefront", the orphaned draft page should be cleaned up. This is a nice-to-have for the initial implementation; document it as a follow-up.

---

## User Stories (High Level)

- As a merchant building a new full-page bundle, I want to preview my bundle on the storefront before it goes live, so I can validate the visual result with confidence.
- As a merchant, I want subsequent preview clicks to be fast, so I can iterate on bundle settings and refresh the preview without waiting for page creation each time.
- As a merchant, I want clicking "Add to Storefront" to seamlessly publish my previewed bundle, so there is no duplication or confusion about which page is live.

---

## Scope

### In Scope
- "Preview" button on the full-page bundle configure page (pre-publish state only).
- Draft Shopify page creation with `shareablePreviewUrl` retrieval.
- `shopifyPreviewPageId` and `shopifyPreviewPageHandle` fields on the Bundle model.
- Prisma migration for new fields.
- Promotion of draft page to published when "Add to Storefront" is clicked.
- New intent `createPreviewPage` in the configure route action.
- New intent `publishPreviewPage` logic integrated into `handleValidateWidgetPlacement`.

### Out of Scope
- Automatic cleanup of orphaned draft pages on bundle deletion (follow-up).
- Preview functionality for product-page bundles (already handled via `onlineStorePreviewUrl`).
- Preview URL sharing / copy-to-clipboard feature.
- Any changes to the Shopify extension (`.liquid`, `.toml`) files.

## Risks

| Risk | Likelihood | Mitigation |
|------|------------|------------|
| Shopify Admin API does not return `shareablePreviewUrl` for pages created with draft flag | Low | Verify in API schema introspection before implementation; `shareablePreviewUrl` is documented for online store resources |
| Draft pages appear in Shopify admin "Pages" list, confusing merchants | Medium | Use a consistent naming convention (e.g., "[Preview] Bundle Name") to distinguish draft previews; document in merchant-facing copy |
| Merchant publishes the draft page manually via Shopify admin before clicking "Add to Storefront" | Low | The `handleValidateWidgetPlacement` promotion step uses `pageUpdate` which is idempotent on an already-published page |
| Draft page metafields go stale if merchant edits bundle settings after preview but before "Add to Storefront" | Medium | The existing `syncBundle` flow already re-writes metafields; add a note in UI copy that preview reflects the last saved state |
