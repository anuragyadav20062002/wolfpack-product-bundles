# Business Requirement: Centralized Auth Layer for API Routes

## Executive Summary

The app's `/api/*` routes have scattered, per-route authentication with no centralized
enforcement layer. Adding a new route without calling `authenticate.*` ships a publicly
accessible endpoint silently. We need a structural pattern that makes authentication
the default, not an opt-in.

---

## Problem Statement

**Who experiences it:** Engineers adding new routes. Security reviewers auditing the app.

**Pain points:**
- 14 routes under `/api/*` each call different auth methods (or none)
- One route (`/api/design-settings/:shopDomain`) already ships with no auth — this is proof the scatter pattern has already caused a miss
- No lint rule or framework enforcement prevents a future auth-less route from shipping
- `/app/*` admin routes ARE protected by a layout route guard (`app/routes/app/app.tsx`), but `/api/*` routes have no equivalent

---

## Research Findings

### Current State of `/api/*` Routes

| Route | Auth Method | Notes |
|-------|-------------|-------|
| `api.billing.{create,cancel,confirm,status}` | `authenticate.admin()` | Admin session required |
| `api.activate-cart-transform` | `authenticate.admin()` | Admin session required |
| `api.check-cart-transform` | `authenticate.admin()` | Admin session required |
| `api.ensure-product-template` | `authenticate.admin()` | Admin session required |
| `api.bundles.json` | `authenticate.public.appProxy()` | Storefront widget calls |
| `api.bundle.$bundleId.json` | `authenticate.public.appProxy()` | Storefront widget calls |
| `api.storefront-products` | None (intentional) | Storefront public |
| `api.storefront-collections` | None (intentional) | Storefront public |
| `api.design-settings.$shopDomain` | **None (accidental)** | ⚠️ Should be checked |
| `api.webhooks.pubsub` | **None** | ⚠️ Internal endpoint, needs shared secret |

**Observation:** Routes split into 3 auth tiers:
1. Admin-authenticated (session-based)
2. App-proxy-authenticated (storefront, Shopify-signed headers)
3. Public/internal (no Shopify session — storefront reads + internal worker)

### What `/app/*` Already Does (The Layout Guard Pattern)

`app/routes/app/app.tsx` has `authenticate.admin(request)` in its loader. Because Remix
runs parent layout loaders before child route loaders, ALL routes nested under `/app/`
inherit this auth check automatically. This is already working correctly.

The gap is that `/api/*` routes are NOT under this layout — they are flat siblings.

### Approach A: Remix Layout Route Guard for `/api/*`

**What it is:** Create `app/routes/api/api.tsx` as a layout route. All routes prefixed
`api.*` become nested children. Put auth logic in the layout loader.

**Problem:** The `/api/*` routes need THREE different auth behaviors, not one. A single
layout loader cannot run `authenticate.admin()` on billing routes AND `authenticate.public.appProxy()`
on storefront routes AND nothing on public storefront routes simultaneously.

Attempts to multiplex (check URL prefix inside the layout) are fragile, bypass the
framework's intent, and must be updated every time a new route is added.

**Verdict:** ❌ Not recommended for `/api/*` — the auth is too heterogeneous.

### Approach B: Sub-layout Route Guards (Tiered)

**What it is:** Create separate layout routes per auth tier:
- `app/routes/api/admin/` → `api.admin.tsx` layout with `authenticate.admin()`
- `app/routes/api/proxy/` → `api.proxy.tsx` layout with `authenticate.public.appProxy()`
- `app/routes/api/internal/` → `api.internal.tsx` layout with shared-secret check
- `app/routes/api/public/` → no auth, intentionally open

Routes are reorganized into these sub-namespaces.

**Pros:**
- Framework-native, idiomatic Remix
- Enforces auth by default for each tier
- New routes automatically inherit the right auth if placed in the correct folder
- No multiplexing logic

**Cons:**
- Requires moving/renaming all 14 API route files
- URL paths change (e.g., `api.billing.create` → `api.admin.billing.create`) unless
  path rewrites are used — which breaks existing callers (widget, Shopify billing callback)
- Large rename commit, higher risk of introducing regressions
- App Proxy URLs are registered in Shopify Partner Dashboard — path changes require
  re-configuration there too

**Verdict:** ❌ Too high blast radius for what we gain. URL path changes break external callers.

### Approach C: Auth Utility Functions (Centralized Helpers, Not Middleware)

**What it is:** Instead of a structural layout change, extract auth into helper functions
that provide a consistent, auditable pattern:

```typescript
// app/lib/auth-guards.server.ts
export async function requireAdminSession(request: Request) { ... }
export async function requireAppProxy(request: Request) { ... }
export async function requireInternalSecret(request: Request) { ... }
```

Each route calls the appropriate helper. The helpers consolidate error handling, logging,
and edge cases in one place. An ESLint custom rule can enforce that every route in
`app/routes/api/` calls one of these helpers (or explicitly opts out with a comment).

**Pros:**
- Zero URL path changes — no external callers broken
- No Remix routing restructure — low risk
- Centralizes auth logic (error messages, logging, edge handling) into one file
- A lint rule can make it a hard failure to ship a route without a guard call
- Works with the existing codebase structure immediately

**Cons:**
- Still opt-in per route (but mitigated by linting)
- Not "structural" enforcement — requires discipline or tooling

**Verdict:** ✅ Recommended — zero blast radius, centralizes logic, lintable.

### Approach D: Express/Node Middleware (True Middleware Layer)

**What it is:** Add auth middleware in `server.ts` that intercepts requests before they
reach Remix route handlers.

**Problem:** This app uses Remix with Vite + the Shopify app template. The `server.ts` is
the Remix Vite dev server wrapper. There is no Express layer to attach middleware to in
the current setup — Remix handles requests directly. Adding Express would require refactoring
the entire server entrypoint and is not compatible with Shopify's Remix app template
deployment model (Cloudflare Workers / Fly.io / standard Node).

**Verdict:** ❌ Not viable without major server architecture change.

---

## Recommended Approach

**Approach C: Auth Utility Helpers + ESLint enforcement**

1. Create `app/lib/auth-guards.server.ts` with `requireAdminSession`, `requireAppProxy`,
   `requireInternalSecret`
2. Migrate all `/api/*` routes to call the appropriate helper
3. Add an ESLint rule (or a simple custom check script) that fails CI if any
   `app/routes/api/*.tsx` file does not call a guard function or have an explicit
   `// auth: public` annotation
4. Fix `api.design-settings.$shopDomain.tsx` — add the missing auth check
5. Fix `api.webhooks.pubsub.tsx` — add `requireInternalSecret`

---

## Functional Requirements

- **FR-01:** Every `/api/*` route must call a typed auth helper that matches its auth tier
- **FR-02:** Auth helpers must centralize error response format (JSON 401/403 for API routes)
- **FR-03:** `requireInternalSecret` must use constant-time comparison to prevent timing attacks
- **FR-04:** A CI check must fail if a new api route ships without calling a guard
- **FR-05:** `api.design-settings.$shopDomain.tsx` must have auth added (shop existence check at minimum)
- **FR-06:** `api.webhooks.pubsub.tsx` must verify `Authorization: Bearer <INTERNAL_WEBHOOK_SECRET>`

## Non-Functional Requirements

- **NFR-01:** No URL path changes — no external callers (widget, Shopify, billing callbacks) are broken
- **NFR-02:** No server architecture changes — stays within Remix Vite + Shopify template
- **NFR-03:** Auth helpers must be typed — `session` and `shop` returned with correct TypeScript types
- **NFR-04:** Zero new dependencies

## Scope

### In Scope
- `app/lib/auth-guards.server.ts` — new centralized auth helpers
- Migration of all `app/routes/api/*.tsx` routes to use helpers
- ESLint/CI enforcement script
- Fix for `api.design-settings.$shopDomain.tsx`
- Fix for `api.webhooks.pubsub.tsx`

### Out of Scope
- `/app/*` admin routes — already protected by layout guard, no change needed
- Auth layout / `app/routes/app/app.tsx` — working correctly, no change
- Cart transform Wasm extension — not a web route
- Server architecture changes

## Risks

| Risk | Likelihood | Mitigation |
|------|------------|------------|
| Migrating routes one-by-one introduces regression | Low | Each helper wraps the same `authenticate.*` call it replaces |
| ESLint rule is too strict and blocks valid public routes | Low | Allow explicit `// auth: public` opt-out annotation |
| INTERNAL_WEBHOOK_SECRET not set in staging/prod | Medium | App startup check logs warning if env var missing |
