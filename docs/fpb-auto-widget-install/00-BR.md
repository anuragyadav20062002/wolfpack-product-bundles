# Business Requirement: FPB Auto Widget Install

## Executive Summary

When a merchant clicks "Add to storefront" on the Full-Page Bundle configure page, the app
should silently install the bundle widget into the active Shopify theme — eliminating the
theme editor step entirely and redirecting the merchant straight to their live storefront page.

## Problem Statement

**Who:** Merchants setting up a full-page bundle for the first time (or on a new store).

**Pain point:** After clicking "Add to storefront", merchants are dumped into the Shopify
Theme Editor with no guidance other than needing to click the "Save" button. They don't need
to configure anything — the widget is pre-configured via page metafields — yet they are
forced through an unfamiliar UI. Many merchants are confused, abandon setup, or contact
support thinking the bundle isn't working.

**Root cause:** The current flow opens a theme editor deep link that _could_ add an app block,
but it relies on the merchant hitting Save. The theme template file
(`templates/page.full-page-bundle.json`) can be written programmatically via the Shopify
Theme REST API, which the app already does for product-page bundles. The FPB flow never
got this treatment.

## Research Findings

### How the widget renders today
- Every FPB page is created with `templateSuffix: full-page-bundle`
- The page metafield `custom.bundle_id` is set by the app at page-creation time
- The Liquid block (`bundle-full-page.liquid`) reads `bundle_id` from the metafield first,
  block settings second — so **no block settings need to be set at install time**
- The template file `templates/page.full-page-bundle.json` is a shared theme asset
  containing the app block; once it exists, ALL FPB pages render automatically

### What the app already knows how to do
`app/services/widget-installation/widget-theme-template.server.ts` contains
`ensureBundlePageTemplate()` — a fully working, idempotent function that:
1. Gets the active (MAIN) theme via GraphQL
2. Checks if `templates/page.full-page-bundle.json` already exists
3. If not, reads `templates/page.json` as a base, injects the app block, and writes
   the template via the Theme REST API (`PUT /themes/{id}/assets.json`)

This is **identical** to what already happens for product-page bundles on every save.
The FPB flow simply never calls it — instead it opens the theme editor.

### Approach A: Keep theme editor, fix previewPath (already done)
- Adds `&previewPath=/pages/{handle}` so the editor opens the correct page
- Merchant still has to click Save
- **Verdict: ❌ Rejected** — still requires an unnecessary manual step

### Approach B: Programmatic install via existing service + redirect to storefront
- `handlePageSelection` calls a new API route `POST /api/install-fpb-widget`
- Server calls `ensureBundlePageTemplate(admin, session, apiKey)` — idempotent write
- On success, client shows a success toast and opens the storefront page directly
- On failure, falls back to opening the theme editor deep link (graceful degradation)
- **Verdict: ✅ Recommended** — zero friction, uses battle-tested existing service

### Approach C: Background install (fire-and-forget on bundle save)
- Auto-install the template whenever a bundle is saved, not just on "Add to storefront"
- Slightly more proactive but silently fails if the session has no theme write scope
- **Verdict: ❌ Rejected** — installs on every save even if merchant never uses storefront;
  the explicit "Add to storefront" button is the right trigger with user intent

## Functional Requirements

- **FR-01:** Clicking "Add to storefront" → selecting a page → triggers a server-side
  template install call; no theme editor is opened
- **FR-02:** Install is idempotent — re-clicking "Add to storefront" on a bundle with the
  template already installed shows a success state, not an error
- **FR-03:** On success, the merchant sees a success toast ("Widget installed!") and the
  storefront page URL is shown/opened so they can preview immediately
- **FR-04:** On failure (theme API error, scope issue, etc.), the flow falls back to opening
  the theme editor deep link with a toast explaining the fallback
- **FR-05:** The install API route must require an authenticated admin session
- **FR-06:** The button label and modal copy should change from "Place Widget" / "theme editor"
  language to "Add to storefront" / "install" language

## Non-Functional Requirements

- **NFR-01:** Install API call should complete within 5 seconds (Theme API is typically <2s)
- **NFR-02:** No DB schema change required — the install is stateless (idempotent asset write)
- **NFR-03:** Zero impact on existing product-page bundle flow
- **NFR-04:** Must work with the existing `SHOPIFY_BUNDLE_BLOCK_UUID` / TOML uid resolution

## User Stories

- As a merchant, I want clicking "Add to storefront" to immediately activate my bundle page,
  so that I don't have to interact with the Shopify Theme Editor
- As a merchant, I want to see a confirmation that my widget is live, so that I know setup
  is complete without navigating away
- As a merchant, if something goes wrong during install, I want a clear fallback path, so
  that I am never left stuck

## Scope

### In Scope
- New `POST /api/install-fpb-widget` API route
- `handlePageSelection` in `route.tsx` — replace theme editor open with API call
- Button/modal copy updates
- Graceful fallback to theme editor on API failure

### Out of Scope
- Product-page bundle flow (already works via `ensureBundlePageTemplate`)
- Auto-install on bundle save (future consideration)
- Multi-theme support (always targets the active/MAIN theme)

## Risks

| Risk | Likelihood | Mitigation |
|------|------------|------------|
| Shopify revokes theme write scopes | Low | App already uses these scopes for PDP bundles |
| Template write fails silently | Low | `writeThemeAsset` throws on non-OK response; caught and returned as error |
| Block UUID not found in TOML | Low | `resolveBlockUuid()` already handles this; fallback to theme editor |
| Merchant's theme incompatible | Very Low | `readBasePageTemplate` falls back to minimal template structure |
