# Business Requirement: DCP Phase 1 Additions

## Executive Summary
The Design Control Panel controls 100+ CSS properties but leaves ~20 visible widget elements hardcoded (search input, skeleton loaders, modal close button, card hover transitions, loading overlay, quantity badge, typography controls, progress bar shape, focus states). Merchants cannot brand these to match their stores.

## Problem Statement
Merchants customise the DCP to match their store's design language. However, when they open the bundle, elements like the search input, skeleton cards, and close button still render with hardcoded colours (#f0f0f0, #777, #e3f2fd) that clash with premium or dark-background themes. These elements were deprioritised in earlier builds but are now consistently flagged in merchant feedback.

## Research Findings
- Existing DCP has 96 typed fields + 5 JSON fields in DesignSettings, stored per-shop per-bundle-type
- CSS variables (--bundle-*) are generated server-side and injected as a `<style>` block; widget CSS uses `var(--bundle-*)` with hardcoded fallbacks
- Hardcoded fallbacks in CSS are the problem — CSS variables for search already exist in the CSS file but are NOT wired to DesignSettings, so they always use the hardcoded fallback
- Widget JS inline styles (loading overlay) bypass CSS entirely; must be converted to CSS-variable-backed values and the widget rebuilt
- Prisma: nullable fields with @default are fully backward compatible — NULL means "use code default"

## Functional Requirements
- FR-01: Search input background, border, focus border, text, placeholder, clear-button colours configurable
- FR-02: Skeleton loading card colours (base + shimmer gradient) configurable
- FR-03: Card hover transition duration and translate-Y lift configurable
- FR-04: Tile quantity badge background and text colour configurable
- FR-05: Modal close button colour, background, hover colour configurable
- FR-06: Loading overlay background and text colour configurable
- FR-07: Button text-transform and letter-spacing configurable
- FR-08: Progress bar height and border-radius configurable
- FR-09: Focus outline colour and width configurable
- FR-10: All new settings persist in DesignSettings per shop per bundle type

## Non-Functional Requirements
- NFR-01: Existing merchants see no visual change — new fields default to current hardcoded values
- NFR-02: Widget CSS/JS rebuild required; no deploy-time schema migration needed (nullable + default)
- NFR-03: Zero ESLint errors introduced

## Scope
### In Scope
All items listed in FR-01 through FR-10.
### Out of Scope
Ad-spend ROAS tracking; checkout UI extension styling (different system); mobile-responsive DCP preview.
