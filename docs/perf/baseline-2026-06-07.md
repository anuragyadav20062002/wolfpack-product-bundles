## Reading the numbers

- **TTFB dominates.** 93 % of LCP is the server response. Phase 4 (loader parallelisation + LRU cache + non-critical work moved to fire-and-forget) is the highest-leverage phase against this.
- **FCP == LCP.** First paint and largest paint land in the same frame — no streaming, no skeleton. Phase 6 (skeleton + Remix `defer()`) will let useful content paint before the heavy data resolves.
- **CLS already fails the fail-budget (0.156 > 0.1).** Most likely cause: dashboard hero image `<img>` tags with no `width`/`height`, plus the post-hydration `new Image()` preload. Phase 3 image work fixes both.
- **Tunnel cost.** A non-trivial slice of TTFB is the `trycloudflare.com` round-trip from Shopify → tunnel → local dev server. Real production over Render's edge typically saves 300–600 ms; budget tuning should wait until we have a sample from production Render.

## Recapture plan

After each phase deploys, re-record the same metrics here under a new H2. Format:

```
## /app/dashboard — after Phase N (YYYY-MM-DD)
| LCP | TTFB | FCP | CLS | INP |
```

Phase 1.7 will additionally aggregate from the `AdminWebVital` table:

```sql
SELECT route, metric,
       COUNT(*) AS samples,
       percentile_cont(0.75) WITHIN GROUP (ORDER BY value) AS p75
FROM "AdminWebVital"
WHERE "shopId" = 'agent-5sfidg3m.myshopify.com'
  AND "createdAt" > now() - interval '24 hours'
GROUP BY route, metric
ORDER BY route, metric;
```
