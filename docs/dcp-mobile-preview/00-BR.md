# Business Requirement: DCP Mobile Preview Toggle

## Executive Summary

Merchants using the Design Control Panel need to preview how their bundle widget looks on mobile viewports without leaving the admin panel, so they can catch layout issues and tune mobile-specific styles before publishing to their storefront.

## Problem Statement

The DCP preview panel renders the bundle widget at a fixed 1440 ├Ś 900 desktop viewport (scaled down to fit the panel), giving merchants no visibility into how the widget renders at mobile screen widths (375px). Mobile commerce accounts for a majority of Shopify traffic; a widget that looks great at 1440px may break at 375px. Currently, the only way to check mobile layout is to test directly on the storefront, which is slow and requires published changes.

## Research Findings

### Approach A: Viewport-mode state inside `PreviewPanel` (change scale divisor)
- Add `viewportMode: "desktop" | "mobile"` state to `PreviewPanel`
- When `mobile`: change the scale base from `DESKTOP_WIDTH (1440)` to a `MOBILE_WIDTH (375)` constant in `StorefrontIframePreview`
- The iframe still renders at full width internally but is scaled as if it were 375px wide ŌĆö the CSS media queries in the widget fire correctly because the iframe's viewport IS 375px
- The existing footer layout toggle buttons serve as the UI template (inline `<button>` style objects already defined in `PreviewPanel.tsx`)
- **Recommended** Ō£ģ

### Approach B: Dual preloaded iframes (one at 1440, one at 375) with opacity crossfade
- Follow the `DualAppPreviewIframe` pattern used for the FPB footer layout toggle
- Two iframes always loaded simultaneously, opacity-toggled on switch
- Pros: instant toggle (no iframe reload)
- Cons: doubles the number of iframes in the modal (4 total for FPB ŌĆö already has dual for sidebar/floating); significant memory overhead; the desktop and mobile iframes must remain sync'd on settings changes
- **Rejected** ŌØī

### Approach C: CSS `transform: scale` on the preview panel container
- Apply a CSS scale to the outer `previewWrapper` div to shrink it to ~375px visual width
- Pros: no iframe changes needed
- Cons: doesn't actually change the viewport inside the iframe ŌĆö CSS media queries in the widget fire based on the iframe's real viewport width, not the scaled visual width; this approach gives a fake mobile appearance without triggering real mobile breakpoints
- **Rejected** ŌØī

### Recommended: Approach A
Changing the `viewportWidth` prop passed to `StorefrontIframePreview` causes the `scale` factor to be computed as `containerWidth / 375` instead of `containerWidth / 1440`. The iframe element's CSS `width` is set to `375px`, so the browser renders it as a genuine 375px viewport ŌĆö the widget's CSS media queries fire correctly. This is the minimal, accurate, and architecturally consistent approach.

## Functional Requirements (FRs)

- FR-01: The DCP preview panel must have a toggle to switch between desktop (1440px) and mobile (375px) viewport simulation
- FR-02: The toggle must be visually consistent with the existing footer layout toggle buttons in `PreviewPanel`
- FR-03: Switching viewport mode must not trigger a page reload or re-fetch ŌĆö it only changes the iframe render width
- FR-04: The viewport toggle must be present for both FPB and PDP bundle types
- FR-05: The currently active viewport mode must be visually indicated (active button state)
- FR-06: Settings changes must be pushed to the iframe via postMessage regardless of current viewport mode (no regression)
- FR-07: The FPB footer layout toggle (sidebar / floating) must continue to work independently from the viewport toggle

## Non-Functional Requirements (NFRs)

- NFR-01: No new network requests ŌĆö viewport switch is purely a render change
- NFR-02: No iframe reload on toggle ŌĆö the iframe's `src` does not change when toggling viewport
- NFR-03: Memory neutral ŌĆö no additional iframes created (single-iframe for PDP; dual-iframe for FPB stays dual)

## User Stories (high level)

- As a merchant, I want to switch to a mobile preview so I can check my bundle looks correct on phones
- As a merchant, I want the toggle to be quick (no reload) so I can iterate on styles rapidly
- As a merchant, I want the desktop preview unchanged so my default workflow is unaffected

## Scope

### In Scope
- `PreviewPanel.tsx` ŌĆö add `viewportMode` state + toggle buttons
- `StorefrontIframePreview.tsx` ŌĆö add `viewportWidth` prop defaulting to `DESKTOP_WIDTH (1440)`; use it for iframe width and scale computation

### Out of Scope
- Tablet viewport simulation
- Responsive breakpoint overlay guides
- Storefront-side mobile testing

## Risks

| Risk | Likelihood | Mitigation |
|------|------------|------------|
| iframe renders content wider than 375px due to widget CSS not having mobile breakpoints | Low | The widget already has responsive CSS; this is a discovery benefit of the feature |
| FPB dual-iframe `DualAppPreviewIframe` needs updating for the viewportWidth prop | Medium | `DualAppPreviewIframe` wraps two `AppPreviewIframe` components ŌĆö pass `viewportWidth` through to both |
| Toggle button styling looks inconsistent with footer layout toggle | Low | Use identical `TOGGLE_BTN_BASE`/`TOGGLE_BTN_ACTIVE` inline style objects already defined in `PreviewPanel.tsx` |
