# Business Requirement: Multi-Condition Step Support

## Executive Summary

Bundle steps currently support only a single condition (e.g. "quantity ≥ 2"), making it impossible for merchants to define a quantity range (e.g. "between 2 and 6 items"). Lower-bound operators (`greater_than`, `greater_than_or_equal_to`) are also broken at the real-time selection layer — they never block additions, so there is no cap on how many items a shopper can add. This feature closes both gaps by storing a second optional condition per step and enforcing both conditions with AND logic in the ConditionValidator.

---

## Problem Statement

### Who is affected
Merchants building bundles with steps that require a minimum quantity (e.g. "pick at least 2 accessories") or a bounded range (e.g. "pick 2–6 snacks").

### Pain points identified

1. **Lower-bound operators are silently broken at selection time.**
   `canUpdateQuantity()` currently returns `{ allowed: true }` for both `greater_than` and `greater_than_or_equal_to` operators — no upper cap is applied, so the shopper can add unlimited items to any step that uses a minimum-quantity condition. The condition is only evaluated at step completion (`isStepConditionSatisfied`), but by that point the shopper may have added 50 items to a "pick at least 2" step.

2. **No bounded-range condition is possible.**
   A merchant who wants "between 2 and 6 items" must choose exactly one condition. They either set `>= 2` (no upper cap) or `<= 6` (no lower cap) — never both.

3. **The UI already supports multiple condition rules per step** (the `addConditionRule` button exists and renders a list), but **only the first rule is saved to the database** — the handler code explicitly reads `stepConditionsData[step.id]?.[0]` and maps it to the single DB columns `conditionType`, `conditionOperator`, `conditionValue`. Any second rule the merchant configures is silently discarded.

---

## Research Findings

### Approach A: Extend DB with two new nullable columns (Recommended)
Add `conditionOperator2` and `conditionValue2` to the `Step` Prisma model (optional, nullable). No `conditionType2` is needed — `conditionType` (quantity vs amount) applies to both conditions on a step, and `ConditionValidator` uses `conditionType` only as a null guard anyway (never branches on the value). The guard for the second condition becomes `conditionOperator2 != null && conditionValue2 != null`.

- **Pros:** Minimal schema change (2 columns, not 3); backward-compatible (new columns nullable); no data migration; handler, API, and ConditionValidator changes are small and targeted; avoids a redundant field
- **Cons:** Doesn't scale beyond 2 conditions (which is sufficient — bounded ranges only need 2 bounds)
- **Verdict:** ✅ Recommended

### Approach B: Store conditions as a JSON array in a single column
Replace the three scalar columns with a `conditions Json?` array column.

- **Pros:** Arbitrary number of conditions
- **Cons:** Requires a data migration to move existing data into the new format; breaks all existing query/update paths that reference the scalar columns; over-engineered for the 2-condition use case; JSON columns don't benefit from DB-level type safety
- **Verdict:** ❌ Rejected — disproportionate migration risk for a 2-condition use case

### Approach C: Compose in ConditionValidator only (no DB change)
Hard-code an upper bound derived from the existing single condition (e.g. if condition is `>= 2`, cap at some default).

- **Pros:** Zero DB changes
- **Cons:** Merchant has no control over the upper bound; coupling between conditions is implicit; doesn't solve the "configure a range" requirement
- **Verdict:** ❌ Rejected — doesn't meet merchant's functional need

## Functional Requirements

- **FR-01:** A merchant can configure up to 2 conditions on a step using the existing condition rule UI (no new UI components).
- **FR-02:** Both conditions, when present, are enforced with AND logic: `condition1 AND condition2`.
- **FR-03:** `canUpdateQuantity()` in ConditionValidator must respect an upper-bound cap derived from either condition (lower-bound operators no longer return unconditionally allowed when a second condition exists).
- **FR-04:** `isStepConditionSatisfied()` must validate that both conditions are met before allowing step navigation.
- **FR-05:** `conditionType` (quantity/amount) applies to both conditions — no `conditionType2` field is needed.
- **FR-06-old → renumbered below:** The second condition is optional; steps with only one condition (or no condition) continue to behave exactly as before.
- **FR-06:** Existing bundle data is fully backward-compatible; no data migration is required.
- **FR-07:** The JSON API endpoints (`api.bundle.$bundleId.json`, `api.bundles.json`) must include the second condition fields so the widget receives them.

## Non-Functional Requirements

- **NFR-01 — Performance:** ConditionValidator logic must remain O(1) — no loops over a dynamic condition list; two explicit condition evaluations.
- **NFR-02 — Backward compatibility:** Any step row with `conditionType2/conditionOperator2/conditionValue2 = NULL` must behave identically to the current implementation.
- **NFR-03 — Testability:** All ConditionValidator multi-condition branches must be covered by unit tests (Jest). Minimum test coverage for multi-condition: lower+upper bound combinations across all operator pairs.
- **NFR-04 — No new UI components:** The existing step condition UI (addConditionRule button + rule list) is the only UI. The second rule the merchant adds flows through to the new DB fields automatically.

---

## User Stories (High Level)

- As a **merchant**, I want to set both a minimum and maximum quantity on a step (e.g. "pick 2–6 items"), so that my bundle enforces a sensible range rather than an open-ended minimum.
- As a **merchant**, I want lower-bound conditions (`>= N`) to actually prevent shoppers from adding unlimited items beyond what I intend, even when I don't set an explicit upper bound.
- As a **shopper**, I want to see a clear message when I try to add more items than a step allows (even if the cap comes from a minimum-quantity condition pair), so that I understand why my action was blocked.

---

## Scope

### In Scope
- Prisma schema: add `conditionOperator2` and `conditionValue2` (nullable) to `Step` — no `conditionType2` needed
- `useBundleConfigurationState`: load both conditions from DB into the rule list
- Full-page configure handler: save second condition rule to the new DB columns
- Product-page configure handler: save second condition rule to the new DB columns
- `api.bundle.$bundleId.json` and `api.bundles.json`: include second condition fields in response
- Cart-transform and metafield-sync types: add second condition fields
- `ConditionValidator`: `canUpdateQuantity` and `isStepConditionSatisfied` — AND-evaluate both conditions
- `_buildLimitText`: produce a combined limit text when both conditions are present (e.g. "2–6 items")
- Unit tests: comprehensive coverage of all multi-condition operator combinations in Jest
- Dashboard handler: include second condition in step data

### Out of Scope
- More than 2 conditions per step
- A dedicated "range" UI (the existing two-rule UI is sufficient)
- Changes to the Prisma `BundlePricingRule` condition system (that's a separate AMOUNT/QUANTITY pricing condition, unrelated)
- Cart-transform Rust/WASM condition enforcement (only ConditionValidator in the JS widget is in scope)

---

## Risks

| Risk | Likelihood | Mitigation |
|------|-----------|------------|
| Prisma migration breaks existing prod data | Low | New columns are nullable; existing rows unaffected |
| `canUpdateQuantity` behavior change for existing single-condition steps | Low | Guard on `conditionType2 != null` before applying second condition |
| Widget receives second condition but old ConditionValidator code doesn't use it | None | Widget JS is rebuilt as part of this feature |
| DB push on SIT first, verify, then push to prod | Low | Standard deploy sequence |
